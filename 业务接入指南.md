## 登录回调链路总览

为避免 OIDC 回调链路混淆，请先统一以下约定：

1. OIDC Provider 的 `redirect_uri` 固定配置到网关：`/auth/callback`
2. 前端回跳地址（frontend callback，建议 `/login/callback`）通过 `/auth/login?callback=...` 传入
3. 网关完成 code 换 token + session 建立后，302 回前端 `<frontend_callback_url>?code=success`

标准链路：

```
Frontend -> /auth/login?callback=<frontend_callback_url>
OIDC Provider -> /auth/callback?code=<auth_code>&state=<state>   (Gateway)
Gateway -> 302 <frontend_callback_url>?code=success
Frontend -> Dashboard
```

若前端误收到原始 `code+state`，应立即转发回 `/auth/callback` 由网关继续处理。

## 概览

Stew 作为业务服务的统一入口，提供以下能力：

```
HTTP/REST Client
       |
       | (JSON over HTTP)
       v
  Stew Gateway
       |-- 认证鉴权（JWT / API Key）
       |-- 注入用户上下文头
       |-- 协议转换（HTTP <-> gRPC）
       |-- 负载均衡
       v
  业务 gRPC 服务
       |
       | gRPC Metadata 中读取用户信息
       v
  业务逻辑
```

业务服务只需：

1. 定义带有 `google.api.http` 注解的 `.proto` 文件
2. 向网关注册服务（携带描述符）
3. 从 gRPC Metadata 读取网关注入的用户信息

## 一、Proto 文件规范

### 1.1 基础文件结构

```protobuf
syntax = "proto3";

// 包名建议使用规范命名
package your.service.v1;

// 如果使用 Go SDK
option go_package = "github.com/your-org/go-sdk/your/service/v1";

// 必须导入：HTTP 注解支持
import "google/api/annotations.proto";

// 可选导入：Stew 认证选项
import "stew/api/v1/options.proto";

// 可选导入：客户端上下文（用于接收结构化用户信息）
import "stew/api/v1/context.proto";
```

### 1.2 HTTP 注解（生成 REST 路由）

通过 `google.api.http` 注解将 gRPC 方法映射为 REST 端点：

```protobuf
service OrderService {

  // GET 请求，路径参数绑定到 proto 字段
  rpc GetOrder (GetOrderRequest) returns (GetOrderResponse) {
    option (google.api.http) = {
      get: "/api/v1/orders/{order_id}"
    };
  }

  // POST 请求，请求体绑定到 proto 字段
  rpc CreateOrder (CreateOrderRequest) returns (CreateOrderResponse) {
    option (google.api.http) = {
      post: "/api/v1/orders"
      body: "*"
    };
  }

  // PUT 请求，混合路径参数与请求体
  rpc UpdateOrder (UpdateOrderRequest) returns (UpdateOrderResponse) {
    option (google.api.http) = {
      put: "/api/v1/orders/{order_id}"
      body: "*"
    };
  }

  // DELETE 请求
  rpc DeleteOrder (DeleteOrderRequest) returns (DeleteOrderResponse) {
    option (google.api.http) = {
      delete: "/api/v1/orders/{order_id}"
    };
  }
}
```

### 1.3 认证控制选项

通过 Stew 扩展选项控制认证行为：

```protobuf
service OrderService {
  // 服务级别：整个服务需要认证
  option (stew.api.v1.auth_reqiured) = true;

  // 继承服务级别：需要认证
  rpc GetOrder (GetOrderRequest) returns (GetOrderResponse) {
    option (google.api.http) = { get: "/api/v1/orders/{order_id}" };
  }

  // 覆盖：此方法不需要认证（公开接口）
  rpc GetOrderStatus (GetOrderStatusRequest) returns (GetOrderStatusResponse) {
    option (google.api.http) = { get: "/api/v1/orders/{order_id}/status" };
    option (stew.api.v1.dont_auth_reqiured) = true;
  }
}

service PublicCatalogService {
  // 服务级别：默认不需要认证
  option (stew.api.v1.auth_reqiured) = false;

  rpc ListProducts (ListProductsRequest) returns (ListProductsResponse) {
    option (google.api.http) = { get: "/api/v1/products" };
  }

  // 覆盖：此管理方法需要认证
  rpc AdminUpdateProduct (AdminUpdateProductRequest) returns (AdminUpdateProductResponse) {
    option (google.api.http) = {
      put: "/api/v1/products/{product_id}/admin"
      body: "*"
    };
    option (stew.api.v1.use_auth) = true;
  }
}
```

**认证优先级（从高到低）：**

```
1. skip_paths 白名单（网关全局配置）
2. dont_auth_reqiured = true（Method 级别豁免）
3. use_auth = true（Method 级别强制）
4. auth_reqiured（Service 级别）
5. 无选项时默认不要求认证
```

### 1.4 AI Guard 字段注解（AI 类接口专用）

如果你的服务提供 AI / LLM 相关接口（如 ChatCompletion、Embedding 等），可以在 Proto 字段上添加 `ai_guard` 注解，让网关自动识别请求体中各字段的语义，从而实现 Token 估算、内容审查、上下文截断等功能。

> 此注解定义在 `stew/api/v1/options.proto` 中的 `AiGuardFieldOptions` 消息，通过 `google.protobuf.FieldOptions` 扩展绑定到字段级别，编号 `50050`。

#### 可用注解字段

| 注解 | 类型 | 说明 |
|------|------|------|
| `is_messages_array` | bool | 标记该字段为消息数组（ChatCompletion 格式的 `messages`） |
| `is_role_field` | bool | 标记该字段为消息中的角色字段（如 `role`） |
| `is_content_field` | bool | 标记该字段为消息中的内容字段（如 `content`） |
| `role_filter` | string | 仅对指定角色的消息进行审查，如 `"user"`（空 = 全部） |
| `is_prompt` | bool | 标记该字段为单一 prompt 字符串（Completion 格式） |
| `is_model` | bool | 标记该字段为模型名称（如 `"gpt-4o"`） |
| `is_max_tokens` | bool | 标记该字段为客户端期望的最大 token 数 |

#### 示例：ChatCompletion 风格接口

```protobuf
syntax = "proto3";
package your.ai.v1;

import "stew/api/v1/options.proto";
import "google/api/annotations.proto";

message ChatMessage {
  string role    = 1 [(stew.api.v1.ai_guard).is_role_field = true];
  string content = 2 [(stew.api.v1.ai_guard).is_content_field = true];
}

message ChatRequest {
  string model                 = 1 [(stew.api.v1.ai_guard).is_model = true];
  repeated ChatMessage messages = 2 [(stew.api.v1.ai_guard) = {
    is_messages_array: true,
    role_filter: "user"
  }];
  int32 max_tokens             = 3 [(stew.api.v1.ai_guard).is_max_tokens = true];
}

message ChatResponse {
  string text = 1;
}

service AiChatService {
  option (stew.api.v1.auth_reqiured) = true;

  rpc Chat (ChatRequest) returns (ChatResponse) {
    option (google.api.http) = {
      post: "/api/v1/ai/chat"
      body: "*"
    };
  }
}
```

#### 示例：简单 Prompt 风格接口

```protobuf
message CompletionRequest {
  string model      = 1 [(stew.api.v1.ai_guard).is_model = true];
  string prompt     = 2 [(stew.api.v1.ai_guard).is_prompt = true];
  int32 max_tokens  = 3 [(stew.api.v1.ai_guard).is_max_tokens = true];
}
```

#### 三层提取策略

AI Guard 按以下优先级从请求体中提取字段：

| 优先级 | 策略 | 来源 | 适用场景 |
|--------|------|------|----------|
| 1 (最高) | Proto FieldOptions | `.proto` 中的 `ai_guard` 注解 | 有 proto 定义的标准 gRPC 服务 |
| 2 | Body Field Map | ETCD 服务配置中的 `body_map` | 非标准字段名或无 proto 的 REST 服务 |
| 3 (兜底) | OpenAI 兼容格式 | 自动识别 `messages` / `prompt` / `model` | 遵循 OpenAI 格式的接口 |

如果你的接口使用非标准字段名且没有 proto 定义，可以在服务注册时通过 `body_map` 指定字段路径映射：

```json
{
  "ai_guard": {
    "enabled": true,
    "body_map": {
      "messages_path": "conversation.history",
      "role_field": "speaker",
      "content_field": "text",
      "user_role_value": "human",
      "model_path": "config.model_name",
      "max_tokens_path": "config.limit"
    }
  }
}
```

## 二、编译描述符文件

网关需要 `.pb` 格式的 `FileDescriptorSet` 来完成协议转换。

### 2.1 编译命令

```bash
# 安装 protoc（如果未安装）
# macOS: brew install protobuf
# Debian/Ubuntu: apt install protobuf-compiler

# 进入 proto 目录
cd your-project/proto

# 编译（需要同时包含所有依赖）
protoc \
  --descriptor_set_out=your_service.pb \
  --include_imports \
  --proto_path=. \
  --proto_path=/path/to/stew/proto \
  --proto_path=/path/to/googleapis \
  your_service.proto
```

### 2.2 使用 Makefile 自动化

```makefile
# Makefile
PROTO_OUT = your_service.pb
PROTO_PATH = proto
STEW_PROTO_PATH = /path/to/stew/proto
GOOGLEAPIS_PATH = /path/to/googleapis

.PHONY: pb
pb:
	protoc \
		--descriptor_set_out=$(PROTO_OUT) \
		--include_imports \
		--proto_path=$(PROTO_PATH) \
		--proto_path=$(STEW_PROTO_PATH) \
		--proto_path=$(GOOGLEAPIS_PATH) \
		$(PROTO_PATH)/your_service.proto

.PHONY: pb-go
pb-go:
	protoc \
		--go_out=generated \
		--go-grpc_out=generated \
		--proto_path=$(PROTO_PATH) \
		--proto_path=$(STEW_PROTO_PATH) \
		--proto_path=$(GOOGLEAPIS_PATH) \
		$(PROTO_PATH)/your_service.proto
```

## 三、服务注册与描述符提交

> 完整接口字段说明与参数表格见附录：[服务注册接口参数完整参考](服务注册接口参数参考.md)

### 3.1 权限模型

Stew 网关采用 **管理端优先** 的服务注册模型，将管理操作与业务侧操作严格分离：

```
                        管理员（前端管理页面）
                               |
                  1. 初始化服务 (InitService)
                               |
                    +----------+-----------+
                    |                      |
              返回 app_id            配置服务属性
              + app_secret          （端点/LB/中间件/健康检查）
                    |
                    v
              业务侧（SDK / CI/CD）
                    |
         使用 app_id 认证，仅允许：
         - 上传/回滚 PB 描述符
         - 可选发送心跳
```

#### 管理端操作（仅管理员）

以下操作只能通过管理端前端页面执行，业务侧无权操作：

| 操作 | 说明 |
|------|------|
| **初始化服务** (`InitService`) | 创建服务，生成 `app_id` + `app_secret` 凭证 |
| **配置端点** | 设置后端地址、端口、权重 |
| **负载均衡** | 选择 LB 算法（RR/WRR/LC/...） |
| **中间件配置** | 限流、CORS、风险评估、Turnstile、AI Guard |
| **健康检查** | 健康探测方式、间隔、阈值 |
| **注销 / 删除服务** | 停止服务或清理记录 |
| **修改服务属性** | 更新版本、标签、元数据等 |

#### 业务侧操作（通过 app_id 认证）

业务服务通过管理员下发的 `app_id`（API Key）认证，仅允许以下操作：

| 操作 | scope | 说明 |
|------|-------|------|
| **上传描述符** | `service:<service_name>:upload_descriptor` | 提交编译好的 `.pb` 文件，触发路由热加载 |
| **回滚描述符** | `service:<service_name>:rollback_descriptor` | 回退到之前的描述符版本 |
| **发送心跳** | `service:<service_name>:heartbeat` | 可选，上报健康状态维持 ETCD 租约 |

> **为什么这样设计？**
> 服务的基础设施配置（端点、负载均衡、中间件策略）属于运维决策，需要人工审核。
> 业务侧只需关注自身 Proto 接口定义的更新，通过 CI/CD 流水线自动提交描述符即可。

#### 只读操作（所有认证用户）

以下查询接口对所有通过认证的用户开放：

- 列出服务 (`ListServices`)
- 查看服务实例 (`GetServiceInstances`)
- 健康检查查询 (`ServiceHealthCheck`)
- 列出描述符 (`ListProtobufDescriptors`)
- 获取描述符 (`GetProtobufDescriptor`)
- 查看版本历史 (`ListDescriptorVersions`)

### 3.2 整体流程

```
1. [管理员] 在管理页面 "注册服务" 向导中完成以下操作：
   a. 填写基本信息（服务名称/描述/协议） -> 系统生成 app_id + app_secret
   b. 上传 .pb 描述符文件（可选,仅 gRPC 服务需要）
   c. 配置端点、负载均衡、健康检查、中间件等属性
   d. 在成功页面复制或下载 app_id/app_secret 凭证
2. [管理员] 将 app_id/app_secret 下发给业务团队
3. [业务侧] 验证路由生效
4. [业务侧] 后续可使用 app_secret 更新描述符
```

网关在接收描述符时会自动执行验证管道：
- 大小检查（上限 10MB）
- 格式验证（确认为有效的 FileDescriptorSet）
- 服务名匹配（描述符中的服务名必须匹配请求的 service_name）
- SHA-256 去重（与当前激活版本比对，相同则幂等跳过）
- 向后兼容性检测（删除的服务/方法/字段类型变更 -> 警告但不阻止）
- `app_id` 凭证校验（验证 scope 和服务绑定关系）

### 3.3 管理端注册服务（向导式流程）

管理员通过前端管理页面的 **注册服务** 向导完成一站式注册：

1. 进入 **服务管理** 页面，点击 **Register new service**
2. **Step 1 - 基本信息**：
   - **服务名称**（必填）：完整的 Protobuf service name，如 `your.service.v1.OrderService`
   - **描述**：服务的简要说明
   - **协议**：`grpc`（默认）或 `http`
   - 提交后系统自动调用 `InitService`，生成 `app_id` 和 `app_secret`
3. **Step 2 - 上传描述符**（可选）：
   - 上传编译好的 `.pb` 描述符文件
   - 系统自动通过 `UploadProtobufDescriptor` 上传并校验
   - 显示发现的 gRPC service 列表
   - HTTP 服务可跳过此步骤
4. **Step 3 - 服务配置**：
   - 配置 endpoint 地址/端口/权重
   - 选择负载均衡策略
   - 配置健康检查端点
   - 配置中间件（限流、CORS、风控、Turnstile、AI Guard）
   - 调用 `RegisterService` 完成注册
5. **Step 4 - 完成**：
   - 显示 `app_id` 和 `app_secret`（仅此一次）
   - 支持一键复制和 JSON 文件下载
   - 点击 "Go to service list" 返回服务管理页面

> `app_id` 本质上是一个 API Key，绑定了对应服务名称和操作 scope，
> 业务侧使用时通过 `x-api-key` 请求头传递。

对应的 gRPC/HTTP 接口（仅管理员可调用）：

```
gRPC: stew.api.v1.ServiceDiscoveryService/InitService
HTTP: POST /api/v1/discovery/services/init
```

### 3.4 业务侧上传描述符

业务侧使用管理员下发的 `app_id` 上传编译好的 `.pb` 描述符文件：

```bash
# 第一步：编译描述符
protoc --descriptor_set_out=order_service.pb --include_imports order_service.proto

# 第二步：上传描述符（使用 app_id 认证）
grpcurl -plaintext \
  -H "x-api-key: $APP_SECRET" \
  -d "{
    \"service_name\": \"your.service.v1.OrderService\",
    \"descriptor_version\": \"v2.1.0\",
    \"descriptor_data\": \"$(base64 -w0 order_service.pb)\",
    \"description\": \"v2.1.0 release\",
    \"previous_version\": \"v2.0.0\"
  }" \
  localhost:3012 stew.api.v1.ServiceDiscoveryService/UploadProtobufDescriptor

# 响应示例：
# {
#   "success": true,
#   "message": "Descriptor uploaded and loaded successfully",
#   "discoveredServices": ["your.service.v1.OrderService"],
#   "compatibilityWarnings": [],
#   "appliedVersion": "v2.1.0",
#   "descriptorHash": "sha256:a1b2c3d4..."
# }
```

上传请求字段说明：

| 字段 | 必填 | 说明 |
|------|------|------|
| `service_name` | 是 | 服务全名（必须与 `app_id` 绑定的服务名一致） |
| `descriptor_version` | 否 | 版本号，为空则自动生成 `{timestamp}-{hash_prefix}` |
| `descriptor_data` | 是 | Base64 编码的 FileDescriptorSet 二进制数据 |
| `description` | 否 | 版本描述 |
| `signature` | 否 | HMAC-SHA256 签名（预留，当前不强制） |
| `force` | 否 | `true` 时忽略兼容性警告 |
| `previous_version` | 否 | 乐观锁：当前激活版本必须等于此值，否则拒绝 |

> **权限校验**：网关会验证 `x-api-key` 对应的凭证是否绑定了请求中的 `service_name`，
> 且 scope 包含 `service:<service_name>:upload_descriptor`。不匹配则返回 403。

### 3.7 使用 Python gRPC 客户端 SDK

Stew 提供了面向 Python 业务服务的高级 gRPC 客户端 SDK，封装了描述符管理和心跳保活的全部底层细节。

> **注意**：SDK 中的 `register()` 和 `deregister()` 方法已废弃。
> 服务注册/注销现在仅通过管理端前端页面操作。

SDK 位于 `/proto/sdk/python/`，依赖已在 `pyproject.toml` 中声明：

```bash
# 使用 uv 安装
cd /path/to/stew/proto/sdk/python
uv sync

# 或通过 pip 直接安装运行时依赖
pip install grpcio protobuf
```

#### 导入说明


#### 服务启动脚本完整示例

将以下脚本作为 Docker `ENTRYPOINT` 或 Kubernetes init-container 中的描述符提交步骤：

```python
#!/usr/bin/env python3
"""submit_descriptor.py — 服务启动时向网关提交描述符"""
import os
import sys
from stew import SyncDiscoveryClient, ConflictError

GATEWAY    = os.environ.get("GATEWAY_ADDR", "127.0.0.1:3012")
APP_SECRET = os.environ["APP_SECRET"]  # 管理员下发的 app_secret
SVC_NAME   = "your.service.v1.OrderService"
PB_PATH    = os.environ.get("DESCRIPTOR_FILE", "order_service.pb")
VERSION    = os.environ.get("SERVICE_VERSION", "latest")

with SyncDiscoveryClient(GATEWAY, app_secret=APP_SECRET) as c:
    # 查询当前激活版本（乐观锁）
    versions = c.list_descriptor_versions(SVC_NAME)
    active = next((v.version for v in versions if v.is_active), None)

    try:
        result = c.upload_descriptor_from_file(
            service_name=SVC_NAME,
            pb_path=PB_PATH,
            version=VERSION,
            description="auto-submitted at startup",
            previous_version=active or "",
        )
        print(f"descriptor applied: {result['applied_version']}")
        for w in result["compatibility_warnings"]:
            print(f"[WARN] {w}", file=sys.stderr)
    except ConflictError as e:
        # 并发启动时其他实例已提交了新版本，幂等忽略
        print(f"[INFO] descriptor already updated by another instance: {e}")
```

#### 乐观锁更新描述符

多实例并发发布时，通过 `previous_version` 保证更新安全：

```python
from stew import DiscoveryClient, ConflictError
import asyncio

async def safe_upload(client: DiscoveryClient, pb_path: str, version: str):
    for attempt in range(3):
        active = await client.get_active_version("your.service.v1.OrderService")
        try:
            result = await client.upload_descriptor_from_file(
                service_name="your.service.v1.OrderService",
                pb_path=pb_path,
                version=version,
                previous_version=active or "",
            )
            return result["applied_version"]
        except ConflictError:
            if attempt == 2:
                raise
            await asyncio.sleep(0.5 * (attempt + 1))
```

#### SDK 错误处理参考

| 异常类 | 触发条件 | 建议处理 |
|--------|----------|----------|
| `DiscoveryError` | gRPC 通信失败或业务拒绝 | 记录日志，视情况重试 |
| `ConflictError` | `previous_version` 乐观锁不匹配 | 重新查询当前版本后重试 |
| `NotFoundError` | 回滚时目标版本不存在 | 先调用 `list_descriptor_versions` 确认版本 |

## 四、用户数据读取

认证通过后，网关会向请求中注入多个 Header，并经由 `HttpToGrpcMetadataMiddleware` 统一转换为 gRPC Metadata，下游业务服务可以直接从 Metadata 中读取，**无需再次验证 Token**。

### 4.1 注入的 Metadata 完整列表

#### 基础用户信息

| Metadata Key | 内容 | 认证来源 | 备注 |
|---|---|---|---|
| `x-user-id` | 用户唯一 ID | JWT `sub` claim / API Key 关联用户 | 主要用户标识 |
| `x-user-email` | 用户邮箱 | JWT `email` claim | 可能为空 |
| `x-user-name` | 显示名 | JWT `preferred_username` 或 `name` claim | 可能为空 |
| `x-token-issuer` | Token 签发方 | JWT `iss` claim | OIDC Provider 地址 |
| `authorization` | `Bearer <access_token>` | Session 模式下重新注入 | 可用于调用其他服务 |

#### API Key 专用

| Metadata Key | 内容 | 备注 |
|---|---|---|
| `x-api-key-id` | API Key 的数据库 ID | UUID 格式 |
| `x-api-key-name` | API Key 名称 | 创建时指定 |
| `x-api-key-scopes` | 权限范围（逗号分隔） | 如 `read,write` |

#### 客户端上下文

| Metadata Key | 内容 | 格式 |
|---|---|---|
| `x-client-context` | 客户端完整上下文 | JSON 字符串 |
| `x-request-id` | 请求追踪 ID | UUID v4 |

```go
    Token   string `json:"token"`
    Tenant  Tenant `json:"tenant"`

    Region  string `json:"region"`

    Sub   string `json:"sub"`
    Name  string `json:"name"`
    Email string `json:"email"`
}

// getMetadataValue 从 gRPC Metadata 中安全读取单值
func getMetadataValue(md metadata.MD, key string) string {
    vals := md.Get(key)
    if len(vals) == 0 {
        return ""
    }
    return vals[0]
}

func (s *OrderServiceServer) GetOrder(ctx context.Context, req *pb.GetOrderRequest) (*pb.GetOrderResponse, error) {
    md, ok := metadata.FromIncomingContext(ctx)
    if !ok {
        return nil, status.Error(codes.Unauthenticated, "missing metadata")
    }

    // 读取基础用户信息
    userID := getMetadataValue(md, "x-user-id")
    userEmail := getMetadataValue(md, "x-user-email")
    userName := getMetadataValue(md, "x-user-name")
    requestID := getMetadataValue(md, "x-request-id")

    if userID == "" {
        return nil, status.Error(codes.Unauthenticated, "missing x-user-id")
    }

    // 读取并解析客户端上下文
    var clientCtx ClientContext
    if ctxJSON := getMetadataValue(md, "x-client-context"); ctxJSON != "" {
        if err := json.Unmarshal([]byte(ctxJSON), &clientCtx); err != nil {
            // 解析失败不中断业务，记录日志即可
            log.Warnf("failed to parse x-client-context: %v", err)
        }
    }

    // API Key 场景
    apiKeyScopes := getMetadataValue(md, "x-api-key-scopes")

    _ = userEmail
    _ = userName
    _ = requestID
    _ = apiKeyScopes
    _ = clientCtx

    // 业务逻辑...
    return &pb.GetOrderResponse{}, nil
}
```

```java
public class OrderServiceImpl extends OrderServiceGrpc.OrderServiceImplBase {

        Metadata.Key.of("x-user-id", Metadata.ASCII_STRING_MARSHALLER);
    private static final Metadata.Key<String> USER_EMAIL_KEY =

        Metadata.Key.of("x-request-id", Metadata.ASCII_STRING_MARSHALLER);

    @Override
    public void getOrder(GetOrderRequest request, StreamObserver<GetOrderResponse> responseObserver) {
        // 从 gRPC 上下文获取 headers（需要拦截器注入）
        Metadata headers = MetadataUtils.getHeadersFromContext();

        String userEmail = headers.get(USER_EMAIL_KEY);

            }
        }

        // 业务逻辑...
        responseObserver.onNext(GetOrderResponse.newBuilder().build());
```

## 五、认证方式配置

### 5.1 选择认证方式

| 场景 | 推荐方式 | 说明 |
|---|---|---|
| Web 前端（浏览器） | OIDC Session Cookie | 自动处理，无需额外配置 |
| SPA / 移动端 | Bearer Token (JWT) | 前端持有 access_token |
| 服务间调用 / 机器账号 | API Key | 稳定、可管理的静态密钥 |
| 混合场景 | jwt_first 策略 | 优先 JWT，失败时尝试 API Key |

### 5.2 调用方携带认证信息

**Bearer Token：**
```http
GET /api/v1/orders/123
Authorization: Bearer eyJhbGciOiJSUzI1NiJ...
```

**Session Cookie（浏览器自动携带）：**
```http
GET /api/v1/orders/123
Cookie: session_id=abc123
```

**API Key - 自定义 Header：**
```http
GET /api/v1/orders/123
x-api-key: your-api-key-here
```

**API Key - Authorization Header：**
```http
GET /api/v1/orders/123
Authorization: Bearer your-api-key-here
```

> 网关通过判断 token 中是否包含两个 `.` 区分 JWT 和 API Key。

>
> - OIDC Provider 的 `redirect_uri` 必须配置为网关地址：`http://<gateway-host>/auth/callback`
> - 前端传给 `/auth/login?callback=...` 的 `callback` 是“登录完成后回前端页面”的地址（建议 `/login/callback`）
> - 网关完成 code 换 token + session 建立后，才会 302 到前端 `callback?code=success`

#### 第三步：回调页面（关键）

```typescript
// pages/Auth/Callback.tsx
import { useEffect, useRef } from 'react';

export default function AuthCallback() {
  const processedRef = useRef(false);

  useEffect(() => {
    // 避免开发模式下 effect 双执行造成重复请求
    if (processedRef.current) return;
    processedRef.current = true;

    const urlParams = new URLSearchParams(window.location.search);
    const code = urlParams.get('code');
    const state = urlParams.get('state');
    const error = urlParams.get('error');
    const errorDescription = urlParams.get('error_description');
    const message = urlParams.get('message');

    // 标准流程：网关已完成 OIDC 回调并设置 HttpOnly Cookie
    if (code === 'success') {
      window.location.href = '/dashboard';
      return;
    }

    // 网关显式返回错误
    if (code === 'error' || error) {
      const msg = message || errorDescription || error || 'Login failed';
      console.error('Login failed:', msg);
      window.location.href = `/login?error=${encodeURIComponent(msg)}`;
      return;
    }

    // 兜底兼容：若误收到原始 OIDC 授权码（code+state），转发回网关处理
    if (code && state) {
      window.location.replace(`/auth/callback?${urlParams.toString()}`);
      return;
    }

    window.location.href = '/login?error=invalid_callback';
  }, []);

  return <div>正在处理登录...</div>;
}
```

### 6.5 认证 API 端点参考

| HTTP 方法 | 路径 | 说明 |
|---|---|---|
| `GET` | `/auth/login?callback=<frontend_url>` | 发起 OIDC 登录，302 跳转到 IDP |
| `GET` | `/auth/callback` | OIDC Provider 回调入口（网关处理 code/state，设置 HttpOnly Cookie） |
| `GET` | `/auth/logout?callback=<url>` | 发起登出，清除 session，302 跳转到 IDP 登出端点 |
| `GET` | `/auth/me` | 获取当前用户信息（依赖 Cookie 认证） |
| `POST` | `/auth/session/validate` | 验证当前 session 有效性 |
| `POST` | `/auth/token/refresh` | 刷新 access_token |

说明：

- OIDC Provider 的 `redirect_uri` 配置在网关侧（例如 `OIDC_REDIRECT_URL=http://localhost:3012/auth/callback`）
- `<frontend_url>` 由业务前端在登录时通过 `callback` 参数传入（建议 `/login/callback`）

回调 URL 格式（网关跳转回前端时附加的参数）：

```
<frontend_callback_url>?code=success
<frontend_callback_url>?code=error&message=<error_message>
```

### 6.6 session 认证机制

```
OIDC 登录完成
        |
        v
网关设置 HttpOnly Cookie（session_id）        <-- 服务端 API 认证使用此 Cookie
        |
        v
302 跳转到前端 /login/callback?code=success
        |
        v
前端检查 code=success，跳转到 /dashboard（无需客户端存储 session_id）
        |
        v
后续请求：浏览器自动携带 HttpOnly Cookie -> 网关验证 session -> 注入用户头到业务服务
```

**安全说明：**
- `HttpOnly Cookie` 无法被 JavaScript 读取，防止 XSS 窃取
- 客户端不存储 session_id，所有认证判断通过服务端 `/auth/me` 验证
- 回调 URL 中不携带 session_id，避免 URL 泄露风险
- 实际 API 安全完全由服务端 Cookie 验证保障

### 6.7 处理 APIResponse 统一响应格式

网关可为接口返回统一的包装格式（通过 `http_response` 选项开启）：

```json
{
  "code": 2000,
  "message": "success",
  "data": { ... }
}
```

`Code` 枚举定义（来自 `stew/api/v1/web.ts`）：

| 枚举值 | HTTP 状态 | 说明 |
|---|---|---|
| `OK = 2000` | 200 | 成功 |
| `PARAMS_ERROR = 4000` | 400 | 请求参数错误 |

## 九、描述符生命周期管理

### 9.1 启动时自动提交描述符

推荐在业务服务的启动脚本或容器 entrypoint 中使用 `app_secret` 自动提交描述符：

> **前提**：管理员已在管理页面 "初始化服务"，并将 `app_secret` 下发给业务团队。

```bash
#!/bin/bash
# entrypoint.sh - 业务服务启动脚本（仅提交描述符，不注册服务）
set -e

GATEWAY_ADDR="${GATEWAY_ADDR:-localhost:3012}"
SERVICE_NAME="your.service.v1.OrderService"
APP_SECRET="${APP_SECRET:?APP_SECRET is required}"
DESCRIPTOR_FILE="${DESCRIPTOR_FILE:-/app/proto/order_service.pb}"
SERVICE_VERSION="${SERVICE_VERSION:-latest}"

# 上传描述符到网关（业务侧唯一的写操作）
echo "Uploading descriptor to gateway..."
UPLOAD_RESPONSE=$(grpcurl -plaintext \
  -H "x-api-key: $APP_SECRET" \
  -d "{
    \"service_name\": \"$SERVICE_NAME\",
    \"descriptor_version\": \"$SERVICE_VERSION\",
    \"descriptor_data\": \"$(base64 -w0 $DESCRIPTOR_FILE)\"
  }" \
  $GATEWAY_ADDR stew.api.v1.ServiceDiscoveryService/UploadProtobufDescriptor 2>&1)

echo "Upload response: $UPLOAD_RESPONSE"

# 检查兼容性警告
WARNINGS=$(echo "$UPLOAD_RESPONSE" | jq -r '.compatibilityWarnings[]? // empty')
if [ -n "$WARNINGS" ]; then
  echo "[WARN] Compatibility warnings detected:"
  echo "$WARNINGS"
fi

echo "Descriptor submitted successfully."

# 启动业务进程（端点、负载均衡等配置由管理员在前端页面完成）
exec "$@"
```

### 9.2 幂等性保证

描述符提交具备天然幂等性：

- 网关对每次上传计算 SHA-256 哈希
- 如果新上传的描述符哈希与当前激活版本相同，直接返回成功，不产生新版本
- 因此业务服务可以在每次启动时无条件提交描述符，无需担心重复

```
第一次启动: upload(v1.pb) -> 新版本 v1.0.0 (hash: abc123)
第二次启动: upload(v1.pb) -> 幂等跳过 (hash unchanged)
滚动更新:   upload(v2.pb) -> 新版本 v2.0.0 (hash: def456)
```

### 9.3 乐观锁（并发安全）

多实例并发更新时，使用 `previous_version` 实现乐观锁：

```bash
# 先查询当前版本
CURRENT=$(grpcurl -plaintext \
  -d '{"service_name": "your.service.v1.OrderService"}' \
  localhost:3012 stew.api.v1.ServiceDiscoveryService/ListDescriptorVersions \
  | jq -r '.activeVersion')

# 上传时指定 previous_version（使用管理员下发的 app_secret）
grpcurl -plaintext \
  -H "x-api-key: $APP_SECRET" \
  -d "{
    \"service_name\": \"your.service.v1.OrderService\",
    \"descriptor_version\": \"v2.1.0\",
    \"descriptor_data\": \"$(base64 -w0 order_service.pb)\",
    \"previous_version\": \"$CURRENT\"
  }" \
  localhost:3012 stew.api.v1.ServiceDiscoveryService/UploadProtobufDescriptor
```

如果在你查询和提交之间有其他实例已经更新了版本，请求会被拒绝（返回版本冲突错误），需要重新查询后重试。

### 9.4 版本回滚

当新版本出现问题时，可以快速回滚到之前的版本：

```bash
# 查看版本历史
grpcurl -plaintext \
  -d '{"service_name": "your.service.v1.OrderService"}' \
  localhost:3012 stew.api.v1.ServiceDiscoveryService/ListDescriptorVersions

# 输出示例：
# {
#   "versions": [
#     {"version": "v2.1.0", "isActive": true, ...},
#     {"version": "v2.0.0", "isActive": false, ...},
#     {"version": "v1.0.0", "isActive": false, ...}
#   ],
#   "activeVersion": "v2.1.0"
# }

# 回滚到 v2.0.0（使用管理员下发的 app_secret）
grpcurl -plaintext \
  -H "x-api-key: $APP_SECRET" \
  -d '{
    "service_name": "your.service.v1.OrderService",
    "target_version": "v2.0.0"
  }' \
  localhost:3012 stew.api.v1.ServiceDiscoveryService/RollbackDescriptor

# 输出示例：
# {
#   "success": true,
#   "message": "Successfully rolled back to version v2.0.0",
#   "activeVersion": "v2.0.0",
#   "discoveredServices": ["your.service.v1.OrderService"]
# }
```

回滚后，网关立即加载目标版本的描述符并重建路由，无需重启。

### 9.5 版本保留策略

- 网关默认保留最近 **5** 个版本
- 后台清理任务每 5 分钟自动清理超出保留数量的旧版本
- 仅清理非激活版本
- 清理不影响正在使用的描述符和已建立的路由

### 9.6 兼容性检测

上传新版本时，网关会自动检测与当前激活版本之间的不兼容变更：

| 检测项 | 示例 | 级别 |
|--------|------|------|
| 服务被删除 | `OrderService` 不再存在 | 警告 |
| 方法被删除 | `GetOrder` 方法被移除 | 警告 |
| 字段类型变更 | `price` 从 `int32` 变为 `string` | 警告 |

所有不兼容变更均作为 `compatibility_warnings` 返回给调用方，**不阻止更新**。业务侧应在 CI/CD 流程中检查这些警告并决定是否继续。

## 十、AI Guard 接入指南（AI 类接口）

如果你的服务提供 AI / LLM 类接口，可以通过 AI Guard 在网关层实现统一的 Token 配额管理、内容审查、上下文截断和审计。

### 10.1 前置条件

1. 网关全局启用 AI Guard（运维配置）：
   ```yaml
   ai_guard:
     enabled: true
     redis_url: "redis://127.0.0.1:6379"
     default_mode: "enforce"
   ```

2. Proto 文件中添加 `ai_guard` 字段注解（参考 [1.4 AI Guard 字段注解](#14-ai-guard-字段注解ai-类接口专用)）

3. 编译并上传描述符文件（描述符中包含字段注解信息）

### 10.2 服务级 AI Guard 配置

通过服务注册接口或管理 UI，为每个服务独立配置 AI Guard 参数。以下是通过 REST 接口注册时携带的配置示例：

```json
{
  "service_name": "your.ai.v1.AiChatService",
  "grpc_address": "127.0.0.1:50051",
  "security": {
    "ai_guard": {
      "enabled": true,
      "mode": "enforce",
      "include_paths": ["/api/v1/ai/"],
      "max_input_tokens": 4096,
      "max_output_tokens": 2048,
      "max_context_tokens": 8192,
      "history_policy": "truncate_to_max_tokens",
      "daily_token_quota": 100000,
      "daily_request_quota": 500,
      "minute_request_quota": 10,
      "quota_window_secs": 86400,
      "allow_free_chat": false,
      "allowed_topics": ["coding", "math", "translation"],
      "deny_keywords": ["ignore previous instructions"],
      "enable_audit": true,
      "classifier_type": "llm",
      "business_description": "A coding assistant that helps users write and debug code.",
      "valid_intent_examples": [
        "How do I implement a binary search in Python?",
        "Fix this Rust lifetime error"
      ],
      "invalid_intent_examples": [
        "Write me a poem",
        "What's the weather today?"
      ]
    }
  }
}
```

#### 关键参数说明

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `enabled` | 是否启用 AI Guard | `false` |
| `mode` | `observe`（仅日志）或 `enforce`（拦截） | `observe` |
| `include_paths` | 要保护的路径前缀列表，空 = 全部 | `[]` |
| `max_input_tokens` | 估算输入 token 上限（0 = 不限） | `0` |
| `max_output_tokens` | 透传给下游的输出 token 预算（0 = 不限） | `0` |
| `max_context_tokens` | 上下文历史 token 上限（0 = 不限） | `0` |
| `history_policy` | `truncate_last_n`：保留最近 N 条；`truncate_to_max_tokens`：按 token 截断 | 无 |
| `daily_token_quota` | 每用户每窗口 token 配额（0 = 沿用全局） | `0` |
| `daily_request_quota` | 每用户每窗口请求配额（0 = 沿用全局） | `0` |
| `minute_request_quota` | 每用户每分钟请求上限（0 = 沿用全局） | `0` |
| `quota_window_secs` | 配额刷新窗口（秒），0 = 沿用全局（默认 86400） | `0` |
| `allow_free_chat` | 是否允许白名单以外的话题 | `false` |
| `allowed_topics` | 允许的话题关键词列表（空 = 不过滤） | `[]` |
| `deny_keywords` | 拒绝关键词黑名单 | `[]` |
| `classifier_type` | `rule`（关键词）或 `llm`（LLM 分类器） | 全局 |
| `business_description` | 服务用途描述，自动注入 LLM 分类器 prompt 作为业务上下文（`llm_system_prompt` 为空时生效） | 无 |
| `valid_intent_examples` | 有效请求的少样本示例列表（正例，用于 few-shot 提示） | `[]` |
| `invalid_intent_examples` | 无关/无效请求的少样本示例列表（负例，用于 few-shot 提示） | `[]` |
| `body_map` | 非标准字段路径映射（见 [1.4](#14-ai-guard-字段注解ai-类接口专用) 三层提取策略） | 无 |
| `endpoint_overrides` | 接口级配置覆盖列表（v2.0，详见 [10.2.1](#1021-接口级配置覆盖v20-新增)） | `[]` |

### 10.2.1 接口级配置覆盖（v2.0 新增）

同一服务下不同 API 端点的成本模型和意图分类需求可能完全不同。例如：
- `/v1/chat/completions` 是高成本端点，需要严格限额和意图分类
- `/v1/embeddings` 是低成本端点，只需宽松限流
- `/v1/models` 是只读端点，根本不需要 AI Guard 保护

通过 `endpoint_overrides` 可以对每个端点独立覆盖服务级参数：

```json
{
  "ai_guard": {
    "enabled": true,
    "mode": "enforce",
    "include_paths": ["/v1/"],
    "daily_token_quota": 100000,
    "minute_request_quota": 30,
    "endpoint_overrides": [
      {
        "endpoint_id": "chat-completions",
        "exact_paths": ["/v1/chat/completions"],
        "minute_request_quota": 5,
        "daily_token_quota": 50000,
        "max_input_tokens": 4096,
        "deny_keywords": ["ignore previous instructions"],
        "classifier_type": "llm",
        "business_description": "A coding assistant for programming questions only."
      },
      {
        "endpoint_id": "embeddings",
        "prefix_paths": ["/v1/embeddings"],
        "minute_request_quota": 60,
        "daily_token_quota": 500000,
        "allow_free_chat": true
      },
      {
        "endpoint_id": "models-list",
        "exact_paths": ["/v1/models"],
        "disabled": true
      }
    ]
  }
}
```

#### 配置继承规则

配置按以下优先级从高到低合并：

```
接口级 (endpoint_overrides 中匹配项的非零/非空字段)
  > 服务级 (ServiceAiGuardConfig)
    > 全局 (ai_guard YAML)
      > 内置默认值
```

**字段覆盖约定**：
- 数值字段（`u32`/`f32`）：非零值覆盖，`0` = 继承上层
- 字符串字段：非空值覆盖，空 = 继承上层
- 列表字段（`Vec<String>`）：非空列表覆盖，空 = 继承上层
- 布尔字段（`disabled`/`allow_free_chat`/`enable_audit`）：使用 Proto3 `optional bool`，未设置 = 继承上层，设置 = 覆盖

#### 端点匹配规则

每条 override 通过 `exact_paths` 和 `prefix_paths` 声明匹配范围。匹配优先级：

1. **精确匹配**（`exact_paths`）优先 -- 请求路径与列表中某项完全相等
2. **最长前缀匹配**（`prefix_paths`）-- 多条匹配时取最长前缀

#### 配额隔离

每个端点使用独立的 Redis 计数器键，格式为 `ai:rl:min:{endpoint_id}:{user_id}:{slot}`。不同端点的配额互不干扰。

#### 关键参数说明

| 参数 | 说明 | 继承规则 |
|------|------|----------|
| `endpoint_id` | 端点标识符（必填，在同一服务内唯一） | -- |
| `exact_paths` | 精确匹配路径列表 | -- |
| `prefix_paths` | 前缀匹配路径列表 | -- |
| `disabled` | 禁用该端点的 AI Guard（`true` = 跳过） | `optional bool`，未设置 = 继承 |
| `mode` | 覆盖运行模式 | 非空覆盖 |
| `minute_request_quota` | 覆盖每分钟限制 | 非零覆盖 |
| `daily_token_quota` | 覆盖 token 配额 | 非零覆盖 |
| `daily_request_quota` | 覆盖请求配额 | 非零覆盖 |
| `deny_keywords` | 覆盖关键词黑名单 | 非空列表覆盖 |
| `allowed_topics` | 覆盖话题白名单 | 非空列表覆盖 |
| `allow_free_chat` | 覆盖自由对话开关 | `optional bool`，未设置 = 继承 |
| `classifier_type` | 覆盖分类器类型 | 非空覆盖 |
| `business_description` | 覆盖业务描述 | 非空覆盖 |
| 其他 LLM/Token 参数 | 同服务级字段 | 非零/非空覆盖 |

### 10.3 网关注入的响应头

AI Guard 处理后会在 HTTP 响应中注入以下头信息，客户端可据此展示配额状态：

| 响应头 | 说明 |
|--------|------|
| `x-ai-guard-action` | 决策结果：`allow` / `deny` / `truncated` / `observed` |
| `x-ai-guard-reason` | 拒绝或截断原因（仅 enforce 模式） |
| `x-ai-guard-endpoint` | 匹配的接口级 endpoint_id（未匹配时不存在） |
| `x-ai-quota-remaining-win-tokens` | 当前窗口剩余 token 配额 |
| `x-ai-quota-remaining-win-requests` | 当前窗口剩余请求次数 |
| `x-ai-quota-win-reset-secs` | 当前配额窗口距重置的剩余秒数 |

此外，网关会向下游 gRPC 服务透传以下请求头：

| 请求头 | 说明 |
|--------|------|
| `x-ai-estimated-tokens` | 估算的输入 token 数 |
| `x-ai-max-tokens` | 输出 token 预算（服务可据此设置 `max_tokens`） |

下游服务可在响应中设置 `x-ai-usage-tokens` 头，告知网关该请求实际消耗的 token 数，用于精确记账。

### 10.4 错误响应

| 情形 | HTTP 状态码 | 说明 |
|------|-------------|------|
| 请求体超限 | 400 | 超过 `request_body_max_bytes` |
| 输入 token 超限 | 400 | 超过 `max_input_tokens` |
| 请求体无法解析 | 400 | enforce 模式下无法识别格式 |
| 关键词命中 | 403 | 匹配 `deny_keywords` |
| 话题不相关 | 403 | 不在 `allowed_topics` 且 `allow_free_chat=false` |
| 配额耗尽 | 429 | 超过 `daily_token_quota` 或 `daily_request_quota` |
| 分钟限额 | 429 | 超过 `minute_request_quota` |

### 10.5 接入检查清单

- [ ] Proto 文件导入了 `stew/api/v1/options.proto`
- [ ] 请求消息中的 AI 字段添加了 `ai_guard` 注解（或配置了 `body_map`）
- [ ] 描述符文件已重新编译并上传
- [ ] 服务注册时设置了 `ai_guard.enabled = true`
- [ ] Redis 已配置并可达（用于配额计数）
- [ ] 确认响应头中出现 `x-ai-guard-action`
- [ ] （可选）下游服务返回 `x-ai-usage-tokens` 实现精确记账
- [ ] （可选）配置了 `endpoint_overrides` 对高成本/低成本接口差异化管控

## 十一、常见问题

**Q: 业务侧如何获取 `app_id` / `app_secret`？**

`app_id` 和 `app_secret` 由管理员在前端管理页面“初始化服务”时生成。业务团队无法自行创建，需联系管理员获取。
`app_secret` 仅在创建时显示一次，请妥善保管。如丢失需联系管理员重新生成。

**Q: 为什么不能直接注册服务了？**

服务的基础设施配置（端点、负载均衡、中间件策略）属于运维决策，需要人工审核。
业务侧只需关注自身 Proto 接口定义的更新（描述符上传），通过 CI/CD 流水线自动提交即可。
`register()`、`deregister()`、`update_service_instance()` 等操作现在仅限管理端通过前端页面执行。

**Q: 业务侧能做哪些操作？**

使用 `app_secret` 认证后，业务侧仅允许：
1. 上传描述符（`UploadProtobufDescriptor`）
2. 回滚描述符（`RollbackDescriptor`）
3. 发送心跳（`UpdateServiceHealth`，可选）

其他操作（注册/注销/删除服务、修改配置）将返回 403 PERMISSION_DENIED。

**Q: 服务注册后，REST 路由没有出现在 OpenAPI 文档中？**

检查顺序：
1. 确认描述符文件已正确上传（`UploadProtobufDescriptor` 接口返回 `success: true`）
2. 确认 `.proto` 中 `google.api.http` 注解语法正确
3. 确认 `package` 和 `service` 名称与注册时的 `service_name` 匹配
4. 访问 `/_openapi/spec.json` 确认路由是否已加载
5. 检查上传响应中的 `discoveredServices` 是否包含预期的服务

**Q: 下游服务收不到 `x-user-id`？**

检查：
1. 接口是否配置了 `auth_reqiured` 或 `use_auth` 选项
2. 客户端请求是否携带了有效的认证凭据（Token 或 API Key）
3. 确认网关日志中没有认证错误（`401 Unauthorized`）
4. 确认 `HttpToGrpcMetadataMiddleware` 在中间件链中已启用（默认已启用）

**Q: 描述符上传报 "Service name mismatch" 错误？**

描述符中发现的服务名必须与请求中的 `service_name` 前缀匹配。检查：
1. 你的 `.proto` 文件中的 `package` + `service` 名是否与 `service_name` 一致
2. 例如 `package your.service.v1;` + `service OrderService` 应匹配 `service_name = "your.service.v1.OrderService"`
3. 使用 `protoc --decode=google.protobuf.FileDescriptorSet` 检查编译后的 `.pb` 内容

**Q: API Key 验证失败，报 `DATABASE_URL not configured`？**

API Key 功能需要 PostgreSQL 支持，请配置环境变量：
```bash
export DATABASE_URL=postgresql://user:pass@localhost:5432/dbname
```
未配置时，API Key 功能不可用，所有 API Key 请求将被拒绝。

**Q: 如何在业务服务中区分 JWT 用户和 API Key 调用？**

检查 `x-api-key-id` 是否存在：
- 存在：API Key 认证，从 `x-api-key-scopes` 读取权限范围
- 不存在：JWT 认证，用户信息来自 `x-user-id` / `x-user-email` 等

```go
// Go 示例
apiKeyID := getMetadataValue(md, "x-api-key-id")
if apiKeyID != "" {
    // API Key 认证流程
    scopes := strings.Split(getMetadataValue(md, "x-api-key-scopes"), ",")
    _ = scopes
} else {
    // JWT 用户认证流程
    userID := getMetadataValue(md, "x-user-id")
    _ = userID
}
```

**Q: `x-client-context` 解析失败？**

`x-client-context` 是 JSON 字符串。解析失败通常发生在：
- 特殊字符导致 Header 值编码问题（极少见）
- 中间件配置不完整，`ClientContextMiddleware` 未启用

解析失败时应记录日志但**不中断业务请求**，`ClientContext` 作为可选的上下文信息处理。

**Q: 描述符上传报 "Version conflict" 错误？**

使用了 `previous_version` 乐观锁，但当前激活版本与指定的 `previous_version` 不匹配。
这通常发生在多个实例并发更新时。解决方法：
1. 重新查询当前激活版本
2. 使用新的 `previous_version` 重试上传
3. 或者不设置 `previous_version`（放弃乐观锁）

**Q: 描述符上传返回 `compatibility_warnings`，是否需要处理？**

兼容性警告不会阻止更新。但建议在 CI/CD 中检查：
- 如果删除了服务或方法，确认调用方已适配
- 如果变更了字段类型，确认上下游序列化兼容
- 设置 `force: true` 可以显式确认已知悉不兼容变更

**Q: 业务服务重启后描述符会丢失吗？**

不会。描述符存储在 ETCD 中（版本化），与服务实例的租约绑定。只要服务实例的 keepalive 正常续约，描述符不会过期。推荐做法是每次启动时都提交描述符（幂等操作），确保即使 ETCD 数据丢失也能恢复。

**Q: 如何回滚到之前的描述符版本？**

```bash
# 查看版本列表
grpcurl -plaintext \
  -d '{"service_name": "your.service.v1.OrderService"}' \
  localhost:3012 stew.api.v1.ServiceDiscoveryService/ListDescriptorVersions

# 回滚到目标版本（使用管理员下发的 app_secret）
grpcurl -plaintext \
  -H "x-api-key: $APP_SECRET" \
  -d '{"service_name": "your.service.v1.OrderService", "target_version": "v2.0.0"}' \
  localhost:3012 stew.api.v1.ServiceDiscoveryService/RollbackDescriptor
```

**Q: 描述符上传大小限制是多少？**

单个描述符文件上限 **10MB**。超过限制将被拒绝。如果业务 proto 非常大，可以考虑拆分为多个服务。

**Q: 如何设置接口请求体限制？**

通过环境变量或 YAML 配置：
```bash
export MAX_REQUEST_BODY_SIZE=10485760  # 10MB（字节）
```

```yaml
server:
  max_request_body_size: "10mb"
```

**Q: AI Guard 启用后响应头中没有 `x-ai-guard-action`？**

检查：
1. 网关全局 `ai_guard.enabled` 是否为 `true`
2. 服务配置中 `ai_guard.enabled` 是否为 `true`
3. 请求路径是否匹配 `include_paths`（空列表 = 所有路径）
4. Redis 是否正常连接（配额检查依赖 Redis）
5. 确认描述符已重新编译并上传（包含 `ai_guard` 字段注解）

**Q: AI Guard 报 400 但请求体格式是正确的？**

字段提取失败的常见原因：
1. Proto 中未添加 `ai_guard` 注解，且 `body_map` 也未配置
2. 字段路径映射错误（如 `messages` 实际在嵌套对象中）
3. `mode = "enforce"` 模式下无法识别格式会直接拒绝，切换到 `"observe"` 可先观察日志

**Q: 如何查看 AI Guard 审计日志？**

启用 `enable_audit: true` 后，每个经过 AI Guard 的请求都会输出结构化审计事件到 tracing 日志，包含 request_id、user_id、endpoint_id、model、estimated_input_tokens、action、reason 等字段。使用 `request_id` 可关联完整的请求链路。`endpoint_id` 仅在命中接口级配置时出现。
