# 认证接入指南

本文档面向业务服务开发者，说明如何在接入 Stew 网关时正确使用认证机制，包括：如何发起认证请求、如何从下游接收用户信息，以及如何通过 Protobuf 选项控制接口的认证行为。

---

## 认证架构概览

```
Client
  |
  | (Cookie session_id / Authorization: Bearer <token> / x-api-key)
  v
Stew Gateway
  |-- UnifiedAuthMiddleware（统一认证中间件）
  |     |-- AuthMiddleware（JWT 认证）
  |     |     |-- SessionAuthStrategy（从 Cookie 获取 session_id）
  |     |     `-- BearerAuthStrategy（从 Authorization Header 获取 Bearer token）
  |     `-- ApiKeyMiddleware（API Key 认证）
  |
  | 认证通过后注入用户信息 Header:
  |   x-user-id, x-user-email, x-user-name, x-token-issuer
  |   authorization (Bearer access_token)
  |
  v
HttpToGrpcMetadataMiddleware
  | 将所有 HTTP Header 转换为 gRPC Metadata
  v
下游 gRPC 业务服务（从 Metadata 读取用户信息）
```

---

## 一、支持的认证方式

### 1. OIDC Session 认证（推荐用于浏览器应用）

适用于通过标准 OIDC 登录流程的 Web 前端。

**流程：**

```
1. 前端跳转到 /auth/login?callback=<frontend_callback_url>
2. OIDC Provider 登录完成后回调到网关 /auth/callback（redirect_uri）
3. 网关完成 code 换 token，创建 session，并设置 Cookie: session_id=<sid>
4. 网关 302 跳转到前端 <frontend_callback_url>?code=success（或 code=error）
5. 后续请求自动携带 Cookie，网关从 SessionStore 获取 access_token 并验证
```

**客户端无需做额外处理**，浏览器会自动携带 Cookie。

### OIDC 回调链路约定（前端必读）

为避免登录回调时序问题，请严格区分以下两个 URL：

- OIDC Provider 的 `redirect_uri`：必须指向网关 `/auth/callback`
- 前端回跳地址（frontend callback）：通过 `/auth/login?callback=...` 传入（建议 `/login/callback`）

推荐配置：

```bash
# 网关侧（OIDC Provider 回调）
OIDC_REDIRECT_URL=http://localhost:3012/auth/callback

# 前端登录入口（传给网关）
/auth/login?callback=http%3A%2F%2Flocalhost%3A5173%2Flogin%2Fcallback
```

标准链路：

```
Frontend -> /auth/login?callback=<frontend_callback_url>
OIDC Provider -> /auth/callback?code=<auth_code>&state=<state>   (Gateway)
Gateway -> 302 <frontend_callback_url>?code=success
Frontend -> 进入业务页面
```

异常兼容建议：

- 若前端回跳页收到 `code + state`（原始授权码中间态），应立即转发到 `/auth/callback`，由网关继续处理
- 前端回跳页应增加一次性执行保护（避免开发模式下 effect 双执行导致重复校验）

---

### 2. Bearer Token 认证（推荐用于前后端分离 / 移动端）

适用于已获取 JWT 的客户端。

**请求 Header：**

```http
Authorization: Bearer <access_token_or_id_token>
```

网关会验证 JWT 签名（通过 OIDC Provider 的 JWKS 端点），并校验 claims（issuer、audience、过期时间）。

---

### 3. API Key 认证（推荐用于服务间调用 / 机器账号）

适用于后台服务、脚本、集成工具等无用户身份的调用场景。

**请求方式一：自定义 Header**

```http
x-api-key: <your-api-key>
```

**请求方式二：Authorization Bearer（非 JWT 格式）**

```http
Authorization: Bearer <your-api-key>
```

> 网关通过判断 token 中是否包含两个 `.` 来区分 JWT 和 API Key。格式为 `header.payload.sig` 的视为 JWT，否则视为 API Key。

---

## 二、认证策略

统一认证中间件支持以下策略，通过环境变量 `AUTH_STRATEGY` 配置：

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| `jwt_first` | JWT 优先，失败后尝试 API Key | 面向用户的混合 API（默认） |
| `api_key_first` | API Key 优先，失败后尝试 JWT | 服务间调用为主 |
| `jwt_only` | 仅允许 JWT 认证 | 纯用户应用 |
| `api_key_only` | 仅允许 API Key 认证 | 纯机器账号场景 |
| `either` | 任意一种通过即可 | 宽松场景 |

**环境变量配置示例：**

```bash
AUTH_STRATEGY=jwt_first        # 默认值
OIDC_CLIENT_ID=your-client-id
OIDC_CLIENT_SECRET=your-secret
OIDC_ISSUER_URL=https://your-idp.example.com
OIDC_REDIRECT_URL=http://localhost:3012/auth/callback
```

说明：`OIDC_REDIRECT_URL` 对应 OIDC Provider 的回调地址，应该始终是网关地址，而不是前端页面地址。

## 三、认证通过后下游可读取的用户信息

认证成功后，网关会将以下信息注入请求 Header，并经由 `HttpToGrpcMetadataMiddleware` 转换为 gRPC Metadata，下游服务可直接读取，**无需再次验证 Token**。

### 注入的 HTTP Header / gRPC Metadata

| Header / Metadata Key | 内容 | 来源 |
|-----------------------|------|------|
| `x-user-id` | 用户唯一 ID（JWT `sub` claim） | JWT / Session |
| `x-user-email` | 用户邮箱 | JWT `email` claim |
| `x-user-name` | 用户名或显示名 | JWT `preferred_username` 或 `name` claim |
| `x-token-issuer` | Token 签发方（JWT `iss` claim） | JWT |
| `authorization` | `Bearer <access_token>`（Session 模式下重新注入） | Session |
| `x-client-context` | 客户端上下文 JSON（IP、UA、设备信息等） | ClientContextMiddleware |

### 在 gRPC 服务端读取示例

**Go（使用 tonic 风格思路 / grpc-go）：**

```go
import "google.golang.org/grpc/metadata"

func (s *MyService) GetUser(ctx context.Context, req *pb.GetUserRequest) (*pb.GetUserResponse, error) {
    md, ok := metadata.FromIncomingContext(ctx)
    if !ok {
        return nil, status.Error(codes.Unauthenticated, "missing metadata")
    }
    userIDs := md.Get("x-user-id")
    if len(userIDs) == 0 {
        return nil, status.Error(codes.Unauthenticated, "missing x-user-id")
    }
    userID := userIDs[0]
    // 使用 userID 进行业务逻辑...
}
```

**Rust（使用 tonic）：**

```rust
use tonic::{Request, Status};

async fn get_user(&self, request: Request<GetUserRequest>) -> Result<Response<GetUserResponse>, Status> {
    let metadata = request.metadata();
    let user_id = metadata
        .get("x-user-id")
        .and_then(|v| v.to_str().ok())
        .ok_or_else(|| Status::unauthenticated("missing x-user-id"))?;
    // 使用 user_id 进行业务逻辑...
}
```

## 四、通过 Protobuf 选项控制认证行为

在 `.proto` 文件中，可以通过 `stew.api.v1` 扩展选项精细控制每个 Service 或 Method 是否需要认证。

### 导入选项定义

```protobuf
import "stew/api/v1/options.proto";
```

### 可用选项

| 选项 | 作用域 | 类型 | 说明 |
|------|--------|------|------|
| `auth_reqiured` | Service | `bool` | 为整个 Service 开启或关闭认证要求 |
| `use_auth` | Method | `bool` | 单独为某个 Method 开启认证（可覆盖 Service 级别设置） |
| `dont_auth_reqiured` | Method | `bool` | 单独为某个 Method 关闭认证（用于 Service 级别开启认证时的豁免） |

### 示例

```protobuf
syntax = "proto3";
package my.service.v1;

import "google/api/annotations.proto";
import "stew/api/v1/options.proto";

// Service 级别：整个服务需要认证
service UserService {
  option (stew.api.v1.auth_reqiured) = true;

  // 继承 Service 级别配置，需要认证
  rpc GetUser (GetUserRequest) returns (GetUserResponse) {
    option (google.api.http) = { get: "/api/v1/users/{id}" };
  }

  // 覆盖：豁免认证（如公开接口）
  rpc GetPublicProfile (GetPublicProfileRequest) returns (GetPublicProfileResponse) {
    option (google.api.http) = { get: "/api/v1/users/{id}/public" };
    option (stew.api.v1.dont_auth_reqiured) = true;
  }
}

// Service 级别：整个服务不需要认证
service HealthService {
  option (stew.api.v1.auth_reqiured) = false;

  rpc Check (HealthCheckRequest) returns (HealthCheckResponse) {
    option (google.api.http) = { get: "/health" };
  }

  // 覆盖：单个方法需要认证
  rpc AdminCheck (AdminCheckRequest) returns (AdminCheckResponse) {
    option (google.api.http) = { get: "/health/admin" };
    option (stew.api.v1.use_auth) = true;
  }
}
```

### 认证控制优先级

```
skip_paths 白名单（最高优先级）
  > dont_auth_reqiured = true（Method 级别豁免）
  > use_auth = true（Method 级别强制）
  > auth_reqiured（Service 级别）
  > 无选项时默认不要求认证（向后兼容）
```

## 五、路径级别跳过认证

除 Protobuf 选项外，也可以通过配置跳过特定路径的认证，适用于健康检查、OpenAPI 文档等公开接口：

```yaml
# config/unified_auth.example.yaml
auth:
  skip_paths:
    - "/health"
    - "/metrics"
    - "/_openapi"
    - "/_openapi/ui"
    - "/_openapi/redoc"
```

或通过环境变量（多值以逗号分隔）：

```bash
AUTH_SKIP_PATHS=/health,/metrics,/_openapi
```

## 六、API Key 管理

### 创建 API Key

调用 `ApiKeyService.CreateApiKey` 接口（需要管理员权限）：

```http
POST /stew.api.v1.ApiKeyService/CreateApiKey
Authorization: Bearer <admin-token>

{
  "name": "my-service-key",
  "scopes": ["read", "write"],
  "expires_at": "2027-01-01T00:00:00Z"
}
```

### 列出 API Key

```http
POST /stew.api.v1.ApiKeyService/ListApiKeys
Authorization: Bearer <admin-token>
```

### 撤销 API Key

```http
POST /stew.api.v1.ApiKeyService/RevokeApiKey
Authorization: Bearer <admin-token>

{ "id": "<key-id>" }
```

> API Key 服务需要配置 `DATABASE_URL` 环境变量指向 PostgreSQL 数据库。未配置时，API Key 验证将返回 stub 错误响应。

## 七、认证错误响应

| HTTP 状态码 | 说明 |
|-------------|------|
| `401 Unauthorized` | Token 缺失、无效、或已过期；Session 过期；API Key 无效 |
| `403 Forbidden` | 认证通过但鉴权（Authorization）拒绝访问 |

**响应体示例：**

```json
{
  "code": 16,
  "message": "Token invalid",
  "details": []
}
```

> `code: 16` 对应 gRPC 状态码 `UNAUTHENTICATED`，网关会将其映射为 HTTP 401。

## 八、常见问题

**Q: Token 验证失败，但 Token 本身是有效的？**  
检查 `OIDC_ISSUER_URL` 是否正确。网关会从 `<issuer>/.well-known/openid-configuration` 拉取 JWKS 进行签名验证，issuer 必须与 Token 中的 `iss` claim 一致。

**Q: 下游服务收不到 `x-user-id`？**  
确认 `HttpToGrpcMetadataMiddleware` 已在中间件链中启用（默认使用 `convert_all()` 模式，会转换所有 `x-` 前缀的 Header）。

**Q: API Key 验证报 "DATABASE_URL not configured"？**  
API Key 功能需要 PostgreSQL 数据库支持，请配置 `DATABASE_URL` 并启动程序（首次启动会自动创建所需表）。

**Q: 创建 API Key 时报 `column "scopes" is of type text[] but expression is of type text`？**  
数据库 `scopes` 列是旧版 `TEXT[]` 类型，需迁移为 `JSONB`。启动程序会自动执行迁移；若需手动修复：
```sql
ALTER TABLE api_keys ALTER COLUMN scopes DROP DEFAULT;
ALTER TABLE api_keys ALTER COLUMN scopes TYPE JSONB USING array_to_json(scopes);
ALTER TABLE api_keys ALTER COLUMN scopes SET DEFAULT '[]';
```

**Q: 如何在不修改代码的情况下临时关闭某接口认证？**  
在 Protobuf 选项中添加 `option (stew.api.v1.dont_auth_reqiured) = true;` 并重新编译描述符，或将该路径加入 `skip_paths` 配置。
