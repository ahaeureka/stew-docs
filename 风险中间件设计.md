# 风险中间件设计

> 返回 [README](../README.zh-CN.md) | 参阅 [中间件](中间件.md) | [安全增强方案](安全增强方案.md) | [配置模型设计](配置模型设计.md)

---

## 目标

设计一个新的请求中间件 `RiskAssessmentMiddleware`，用于在网关中统一消费 CDN/WAF 与外部情报源标签，并输出统一的风险动作。

该中间件主要解决以下问题：

1. 统一接收国家、ASN、机房、代理、VPN、Tor、Bot 分数等风险字段。
2. 将上游风险信号转化为统一动作：`allow`、`observe`、`throttle`、`challenge`、`block`。
3. 在匿名高价值入口上与 Cloudflare Turnstile 联动。
4. 为 OPA/Casbin、限流和审计提供统一的风险上下文。

本设计默认风险中间件开关和参数按注册服务绑定：

- 配置来源不是全局 `AppConfig.security.risk`。
- 配置来源是 `ServiceInstance.middleware_config.risk`。
- 配置随服务注册或编辑写入 ETCD。
- 网关基于 `service_name` 从 ETCD 缓存获取对应服务的风险策略。

---

## 中间件位置

推荐顺序如下：

```text
RequestId
-> RequestIdSpan
-> ClientContext
-> RiskAssessment
-> Turnstile
-> UnifiedAuth
-> RateLimit
-> HttpToGrpcMetadata
-> Logging
```

原因：

- `ClientContext` 先完成来源提取，风险中间件才能使用结构化来源数据。
- `RiskAssessment` 应在认证前执行，优先拦截匿名高风险流量。
- `Turnstile` 应根据风险动作或路径要求触发，而不是无差别执行。

补充说明：

- 中间件可以全局注册到链路中。
- 但它对某个请求是否实际生效，要看该请求映射到的注册服务是否启用了 `middleware_config.risk_enabled`。

---

## 设计边界

`RiskAssessmentMiddleware` 负责：

- 读取和标准化风险标签。
- 根据静态配置或策略输出风险动作。
- 将风险结果注入请求头或 metadata，供后续中间件使用。
- 对 `block` 动作直接返回错误。
- 按 `service_name` 选择对应的服务级风险配置。

`RiskAssessmentMiddleware` 不负责：

- 自行调用外部 GeoIP 或情报源接口做实时查询。
- 直接实现复杂 Bot 识别引擎。
- 替代 OPA/Casbin 的完整策略系统。
- 维护独立于服务注册之外的另一套运行时配置存储。

## 输入

### 1. 请求头输入

中间件需要消费来自上游的头，包括但不限于：

- 客户端真实 IP。
- 国家、地区。
- ASN。
- 数据中心或托管标记。
- 代理、VPN、Tor 标记。
- Bot 分数。
- WAF 命中规则 ID。
- 边缘处置动作标记。

### 2. 路径与服务信息

中间件需要读取：

- HTTP path。
- HTTP method。
- 若能解析出 gRPC service/method，则作为后续规则匹配条件。

### 3. 本地配置输入

中间件消费服务实例上的 `middleware_config.risk` 配置，包括：

- 全局开关。
- challenge 路径。
- 高风险国家。
- 代理、机房、Tor 等控制开关。
- 动作覆盖规则。

这里的全局开关指的是某个服务内的默认风险策略，而不是整个网关的全局开关。

### 4. 策略输入

如后续接入 OPA/Casbin，可将下列上下文传入策略层：

- `risk.country`
- `risk.region`
- `risk.asn`
- `risk.datacenter`
- `risk.proxy`
- `risk.vpn`
- `risk.tor`
- `risk.bot_score`
- `risk.waf_rule_id`
- `risk.path_class`

## 输出

中间件统一输出以下动作之一：

```rust
pub enum RiskAction {
    Allow,
    Observe,
    Throttle,
    Challenge,
    Block,
}
```

对应含义：

- `Allow`：继续处理。
- `Observe`：继续处理，但记录日志与指标。
- `Throttle`：继续处理，并给后续限流层提供加强限流信号。
- `Challenge`：要求后续 Turnstile 中间件执行挑战。
- `Block`：直接拒绝。

## 推荐的数据结构

### 1. 风险上下文

```rust
#[derive(Debug, Clone, Default, Serialize, Deserialize)]
pub struct RiskContext {
    pub client_ip: String,
    pub country: Option<String>,
    pub region: Option<String>,
    pub asn: Option<String>,
    pub datacenter: Option<bool>,
    pub proxy: Option<bool>,
    pub vpn: Option<bool>,
    pub tor: Option<bool>,
    pub bot_score: Option<u32>,
    pub waf_rule_id: Option<String>,
    pub waf_action: Option<String>,
    pub action: RiskAction,
    pub matched_rule: Option<String>,
    pub reasons: Vec<String>,
}
```

### 2. 中间件结构

```rust
pub struct RiskAssessmentMiddleware {
  service_discovery: Arc<EtcdServiceDiscovery>,
  trusted_proxy: Arc<TrustedProxyConfig>,
}
```

说明：

- 风险配置不建议在中间件启动时一次性固化。
- 应按请求解析 `service_name` 后，从 `EtcdServiceDiscovery` 缓存读取对应实例的 `middleware_config.risk`。

## 决策流程

建议流程如下：

1. 读取请求 path、method、headers。
2. 从 path 中提取或解析目标 `service_name`。
3. 通过 `EtcdServiceDiscovery` 查询该服务实例缓存。
4. 读取实例上的 `middleware_config.risk_enabled` 和 `middleware_config.risk`。
5. 若该服务未启用风险中间件，则直接放行。
6. 从可信头中提取国家、地区、ASN、代理、VPN、Tor、Bot 分数等字段。
7. 构建 `RiskContext`。
8. 先执行该服务自己的静态规则匹配。
9. 若启用了 OPA/Casbin 风险联动，则将 `RiskContext` 传给策略层补充判断。
10. 生成最终 `RiskAction`。
11. 将结果写入请求头，例如 `x-risk-action`、`x-risk-reasons`、`x-risk-country`。
12. 若动作为 `block`，直接返回 `403 Forbidden`。
13. 若动作为 `challenge`，标记请求，由后续 Turnstile 中间件执行挑战。
14. 若动作为 `throttle`，为后续限流中间件提供更严格阈值依据。

## 推荐的初版规则

建议第一版先做规则式判断，不一开始就引入复杂策略引擎。

所有规则均以当前请求所属服务的 `middleware_config.risk` 为准，不跨服务共享启停状态。

### 1. 路径优先规则

- 命中 `challenge_paths` 的匿名入口，默认输出 `challenge`。
- 命中 `observe_only_paths` 的接口，仅输出 `observe`。

### 2. 来源规则

- 高风险国家命中时，匿名入口优先 `challenge`。
- Tor 命中时，默认 `block` 或 `challenge`。
- 代理/VPN 命中时，对匿名入口优先 `challenge`。
- 数据中心流量命中时，优先 `observe` 或 `throttle`，不建议初版直接 `block`。

### 3. Bot 分数规则

- `bot_score <= threshold` 且命中高价值匿名入口时，输出 `challenge`。
- 极低 Bot 分数且命中重复失败来源时，可升级为 `block`。

### 4. WAF 命中规则

- 若边缘已经明确给出 `block` 动作，则网关直接拒绝。
- 若边缘给出 `managed_challenge` 或同类标记，则网关可降级为 `challenge` 或 `observe`，避免重复封禁。

## 与 Turnstile 的联动方式

推荐不要让 `RiskAssessmentMiddleware` 自己直接调用 Turnstile，而是通过请求标记驱动后续中间件。

建议注入的请求头：

- `x-risk-action: challenge`
- `x-risk-reasons: proxy,high_risk_country`
- `x-risk-country: XX`
- `x-risk-bot-score: 12`

后续 `TurnstileMiddleware` 可以扩展为：

- 如果 path 在 `turnstile.skip_paths` 中，仍可跳过。
- 如果当前服务 `middleware_config.turnstile_enabled=false`，则不执行 challenge。
- 如果存在 `x-risk-action=challenge`，则强制要求 token。
- 如果 path 本身属于当前服务的 `middleware_config.turnstile.required_paths`，也要求 token。

---

## 与限流的联动方式

建议 `RateLimitMiddleware` 后续支持读取风险信号：

- `x-risk-action=throttle`
- `x-risk-proxy=true`
- `x-risk-bot-score=<n>`

然后应用更严格阈值，例如：

- 匿名代理流量阈值降为全局阈值的 30%。
- challenge 失败来源进入临时更严阈值。
- 高价值路径使用单独路径限流键。

该联动也应遵循现有模式：

- `RateLimitMiddleware` 继续从 ETCD 中读取服务级 `middleware_config`。
- 风险中间件只输出信号，不自己持有限流配置。

---

## 与 OPA/Casbin 的联动方式

建议把 `RiskContext` 视为授权上下文的一部分，而不是独立平行系统。

可传入的策略输入示例：

```json
{
  "path": "/auth/login",
  "method": "POST",
  "authenticated": false,
  "risk": {
    "country": "XX",
    "proxy": true,
    "tor": false,
    "datacenter": true,
    "bot_score": 18,
    "waf_rule_id": "cf-managed-bot"
  }
}
```

策略层职责：

- 决定匿名入口是否 `challenge`。
- 决定后台入口是否直接 `block`。
- 决定认证用户请求是 `allow` 还是 `throttle`。

## 错误响应设计

### 1. Block

- HTTP 状态码：`403 Forbidden`
- 响应体：简洁描述，不泄露过多规则细节。

建议格式：

```text
Request blocked by gateway risk policy.
```

### 2. Challenge

- 不由风险中间件直接返回。
- 由 Turnstile 中间件在 token 缺失、无效、失败时返回对应 `400/403`。

## 可观测性

每次风险动作至少记录以下字段：

- `request_id`
- `path`
- `method`
- `client_ip`
- `country`
- `asn`
- `proxy`
- `vpn`
- `tor`
- `datacenter`
- `bot_score`
- `risk_action`
- `matched_rule`
- `risk_reasons`

建议指标：

- `gateway_risk_action_total{action=...}`
- `gateway_risk_challenge_total`
- `gateway_risk_block_total`
- `gateway_risk_proxy_hits_total`
- `gateway_risk_country_hits_total{country=...}`

## 降级策略

### 1. 上游风险头缺失

- 默认退化为 `allow` 或 `observe`，不直接阻断。
- 记录一次数据缺失日志。

### 2. 配置缺失

- 若当前服务不存在 `middleware_config` 或 `risk_enabled=false`，中间件为 no-op。

### 3. 策略层不可用

- 静态规则继续生效。
- OPA/Casbin 补充决策可降级为跳过。

### 4. 外部情报源不可用

- 不建议由请求链同步调用外部情报源。
- 如必须使用，应通过缓存和离线同步方式降低影响。

---

## 实现顺序建议

1. 在 `service_discovery.proto` 的 `ServiceMiddlewareConfig` 中扩展风险字段。
2. 在服务注册和服务编辑链路中支持写入新的风险配置到 ETCD。
3. 在 `middleware/mod.rs` 中注册 `risk_assessment.rs`。
4. 实现初版基于 ETCD 服务缓存和静态规则的 `RiskAssessmentMiddleware`。
5. 在 `MiddlewareConfigurator` 中将其插入 `ClientContext` 与 `Turnstile` 之间。
6. 为 Turnstile 中间件增加基于服务级 `middleware_config.turnstile` 和 `x-risk-action` 的强制挑战模式。
7. 为 RateLimit 中间件增加基于风险标记的差异化限流能力。
8. 再视需要接入 OPA/Casbin 联动。

---

## 验收标准

- 风险中间件能按 `service_name` 从 ETCD 缓存读取对应服务的风险配置。
- 风险中间件能读取并标准化上游风险头。
- 能输出统一动作并和 Turnstile、限流产生联动。
- `block`、`challenge`、`observe`、`throttle` 四类动作均有日志可观测。
- 中间件失效或缺少风险头时，系统有明确降级行为。
