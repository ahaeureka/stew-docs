## 登录回调链路总览

部署前先确认 OIDC 回调链路配置正确：

1. `OIDC_REDIRECT_URL` 必须指向网关 `/auth/callback`
2. 前端回跳页通过 `/auth/login?callback=<frontend_url>` 传入
3. 网关处理完成后，再 302 到前端 `code=success`

快速自检：

```bash
# 正确
OIDC_REDIRECT_URL=https://gateway.example.com/auth/callback

# 错误（不要这样）
OIDC_REDIRECT_URL=https://frontend.example.com/login/callback
```

## 启动场景

### 场景 1：最小化启动（无外部依赖）

仅启动网关，使用本地服务，不连接 gRPC 后端或 ETCD。

```bash
export LISTEN_ADDR=127.0.0.1:3012
cargo run --release
```

此时：
- 本地注册的服务（认证/授权/服务发现）正常工作
- 远程 gRPC 代理不可用（无后端）
- OpenAPI 文档可用
- 后台任务会尝试 gRPC 重连

### 场景 2：连接 gRPC 后端

```bash
export LISTEN_ADDR=127.0.0.1:3012
export GRPC_BACKEND_URL=http://127.0.0.1:50051
cargo run --release
```

此时远程 gRPC 代理可用，请求按 LocalFirst 策略路由。

### 场景 3：启用 ETCD 服务发现

```bash
# 先启动 ETCD
docker compose -f docker-compose.etcd-dev.yml up -d

export LISTEN_ADDR=127.0.0.1:3012
export ETCD_ENABLED=true
export ETCD_ENDPOINTS=http://127.0.0.1:2379
cargo run --release
```

此时：
- 服务注册/发现通过 ETCD
- 描述符热加载与路由热重载生效
- 分布式路由冲突检测启用

### 场景 4：启用 TLS

```bash
export LISTEN_ADDR=0.0.0.0:443
export TLS_ENABLED=true
export TLS_CERT_PATH=/app/certs/server.crt
export TLS_KEY_PATH=/app/certs/server.key
cargo run --release
```

启用 mTLS：
```bash
export TLS_REQUIRE_CLIENT_AUTH=true
export TLS_CLIENT_CA_PATH=/app/certs/ca.crt
```

### 场景 5：启用认证

```bash
export OIDC_CLIENT_ID=your-client-id
export OIDC_CLIENT_SECRET=your-client-secret
export OIDC_ISSUER_URL=https://your-oidc-provider.com
export OIDC_REDIRECT_URL=http://localhost:3012/auth/callback
export AUTH_STRATEGY=jwt_first
cargo run --release
```

> OIDC 回调地址防误配（上线前必查）
>
> - `OIDC_REDIRECT_URL` 必须配置为网关地址 `/auth/callback`
> - 不要配置成前端页面地址（如 `/login/callback` 或 `/auth/callback` 的前端域名）
> - 前端回跳地址应通过 `/auth/login?callback=<frontend_url>` 传入，而不是写在 `OIDC_REDIRECT_URL`

正确示例：

```bash
OIDC_REDIRECT_URL=https://gateway.example.com/auth/callback
```

错误示例（不要这样配）：

```bash
OIDC_REDIRECT_URL=https://frontend.example.com/login/callback
```

常见异常表现（通常由误配触发）：

- 前端回调页直接收到原始 `code+state` 中间态
- 登录回调报错（如 Unknown error）或首次校验偶发失败
- 需要手动重试后才能进入系统

### 场景 6：启用 PostgreSQL 持久化

```bash
# 先启动 PostgreSQL
docker compose -f docker-compose.yml up postgres -d

export DATABASE_ENABLED=true
export DATABASE_URL=postgresql://stew:password@localhost:5432/stew_db
cargo run --release
```

### 场景 7：YAML 配置文件启动

```bash
export CONFIG_PATH=/app/config.yaml
cargo run --release
```

## 数据库初始化

网关使用 PostgreSQL 存储以下数据：
- **API Keys**（`api_keys` 表）：API 密钥管理
- **用户会话**（`user_sessions` 表）：OIDC 会话状态
- **审计日志**（`audit_logs` 表）：认证/授权审计记录
- **服务实例**（`service_instances` 表）：ETCD 服务注册的持久化备份（可选）
- **分布式锁**（`distributed_locks` 表）：分布式协调（可选）

### 环境配置优先级

> **重要**：遵循"不在代码中硬编码密码"原则，数据库密码通过环境变量传递。

配置加载顺序：
1. 系统环境变量（最高优先级）
2. `.env.{RUST_ENV}` 文件（如 `.env.production`）
3. `.env` 基础配置文件
4. 代码默认值（最低优先级）

**环境变量示例**：
```bash
# 运行环境（development / production / staging）
RUST_ENV=development

# 主数据库连接（必需）
DATABASE_URL=postgresql://stew_dev:your_password@localhost:5432/stew_dev

# Casbin 策略数据库（可选，为空时使用本地 CSV 文件）
CASBIN_DATABASE_URL=postgresql://stew_dev:your_password@localhost:5432/stew_dev
```

**文件组织**：
| 文件 | Git 跟踪 | 用途 |
|------|----------|------|
| `.env` | **忽略** | 开发环境真实密钥（本地使用） |
| `.env.production` | **忽略** | 生产环境真实密钥（部署时创建） |
| `.env.staging` | **忽略** | 预发布环境密钥 |
| `.env.example` | 跟踪 | 模板文件，含占位符，无真实密钥 |
| `config.production.yaml` | 跟踪 | 使用 `${DATABASE_URL}` 占位符 |

### 数据库迁移脚本（可选）

为方便部署，可以创建一键初始化脚本：

```bash
#!/bin/bash
# scripts/init_database.sh

set -e

DB_USER="${DB_USER:-stew_dev}"
DB_PASSWORD="${DB_PASSWORD:-1q2w3e!*\$Stew_dev}"
DB_NAME="${DB_NAME:-stew_dev}"
DB_HOST="${DB_HOST:-localhost}"

echo "🔧 Initializing database: $DB_NAME"

# 创建用户和数据库
psql -U postgres -h $DB_HOST <<EOF
CREATE USER $DB_USER WITH PASSWORD '$DB_PASSWORD';
CREATE DATABASE $DB_NAME OWNER $DB_USER;
GRANT ALL PRIVILEGES ON DATABASE $DB_NAME TO $DB_USER;
EOF

# 执行迁移
PGPASSWORD="$DB_PASSWORD" psql -h $DB_HOST -U $DB_USER -d $DB_NAME \
  -f migrations/001_initial_schema.sql

PGPASSWORD="$DB_PASSWORD" psql -h $DB_HOST -U $DB_USER -d $DB_NAME \
  -f migrations/001_create_service_registry_schema.sql

echo "✅ Database initialized successfully"
```

**使用方式**：
```bash
chmod +x scripts/init_database.sh

# 开发环境
./scripts/init_database.sh

# 生产环境（传递自定义参数）
DB_USER=stew_prod DB_PASSWORD='YourStrongPassword' DB_NAME=stew_prod DB_HOST=prod-db-host \
  ./scripts/init_database.sh
```

### 常见问题

**Q: 启动时报 `no such column: api_keys.key_hash`**  
A: 数据库表是旧版本创建的，缺少 `key_hash` 列。解决方案：
```sql
-- 手动添加该列
ALTER TABLE api_keys ADD COLUMN IF NOT EXISTS key_hash VARCHAR(64) NOT NULL DEFAULT '';
CREATE UNIQUE INDEX IF NOT EXISTS idx_api_keys_key_hash ON api_keys(key_hash) WHERE key_hash != '';
```
或者重新执行程序的自动迁移（会自动检测并添加缺失列）。

**Q: 启动时报 `column "scopes" is of type text[] but expression is of type text` 或 `column "metadata" is of type jsonb but expression is of type text`**  
A: 旧版本将 `scopes` 列定义为 `TEXT[]`，新版本改为 `JSONB` 存储。程序启动会自动迁移，但如果自动迁移未生效，可手动执行：
```sql
-- 迁移 scopes 列：TEXT[] -> JSONB
ALTER TABLE api_keys ALTER COLUMN scopes DROP DEFAULT;
ALTER TABLE api_keys ALTER COLUMN scopes TYPE JSONB USING array_to_json(scopes);
ALTER TABLE api_keys ALTER COLUMN scopes SET DEFAULT '[]';
```

**Q: 如何在不设置 `DATABASE_URL` 的情况下启动？**  
A: 程序会自动降级为 SQLite 内存数据库：
```
DATABASE_URL not set, using default SQLite database: /tmp/stew_apikeys.db
```
此模式适合快速开发测试，但 API Key 和会话数据不会持久化。

**Q: `config.production.yaml` 中能直接写密码吗？**  
A: **不可以**。该文件已被 git 跟踪，绝不能写入真实密码。正确做法：
```yaml
database:
  url: "${DATABASE_URL}"  # 占位符，从环境变量读取
```

**Q: 生产环境如何管理密钥？**  
A: 推荐方案：
1. **容器化部署**：通过 Docker/Kubernetes 的 Secret 注入环境变量
2. **密钥管理工具**：HashiCorp Vault / AWS Secrets Manager
3. **CI/CD 流水线**：在部署时动态注入 `DATABASE_URL`
4. **不使用 .env 文件**：直接在系统环境变量中配置（最高优先级）

```
/services/{service_name}/{instance_id}    -- 服务实例（二进制 Protobuf）
```

```bash

```

## 故障排查

### 启动失败

| 症状 | 原因 | 解决 |
|------|------|------|
| `TLS_CERT_PATH not configured` | TLS 启用但未配置证书 | 设置 `TLS_CERT_PATH` 和 `TLS_KEY_PATH` |
| ETCD 连接超时 | ETCD 不可达 | 检查 `ETCD_ENDPOINTS` 和网络连通性 |
| 描述符加载报错 | `.pb` 文件不存在/损坏 | 检查 `proto/*.pb` 文件路径和内容 |

### 请求路由问题

| 症状 | 原因 | 解决 |
|------|------|------|
| 404 Not Found | REST 路由未注册 | 检查 proto 中 `google.api.http` 注解 |
| 503 Service Unavailable | 无可用后端 | 检查 `GRPC_BACKEND_URL` 或 ETCD 服务状态 |
| 本地服务响应但远程不可用 | LocalFirst 策略正常 | 检查远程后端连接 |

### 认证问题

| 症状 | 原因 | 解决 |
|------|------|------|
| 401 Unauthorized | Token 无效或过期 | 检查 JWT/API Key |
| OIDC 初始化失败 | Issuer URL 不可达 | 检查 `OIDC_ISSUER_URL` |
| API Key 不可用 | 数据库未配置 | 设置 `DATABASE_URL` 或禁用 API Key |

### 日志调试

```bash

export RUST_LOG=stew=debug

# 开启 trace 级别（含请求/响应详情）
```
