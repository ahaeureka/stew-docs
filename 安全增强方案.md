# 安全增强方案

---

## 目标

结合当前 Stew 网关已有能力，落地以下四类边界安全增强：

1. 收紧 CORS 白名单。
2. 接入 CDN/WAF Bot 管理与基础风控。
3. 对匿名高价值入口增加人机验证。
4. 对异常国家、机房、代理流量做灰度拦截。

本文档给出的是基于当前实现的改进方案，不假设重写网关核心架构，优先复用现有中间件链、客户端上下文提取、限流与授权能力。

补充约束：以下安全能力建议按注册服务绑定，而不是以网关全局开关统一控制。

- 配置承载位置优先使用 `ServiceInstance.middleware_config`。
- 配置通过服务注册或服务编辑写入 ETCD。
- 网关运行时通过 ETCD watch 热更新对应服务的安全策略。
- 仅可信代理头、Turnstile 密钥、底层 HTTP 客户端参数这类平台能力保留为全局配置。

---

## 现状评估

当前网关已经具备以下可复用能力：

- 响应侧 CORS 中间件，支持自定义来源、方法、头部白名单。
- 请求侧 ClientContext 中间件，可提取客户端 IP、UA、Origin，以及上游注入的国家和地区信息。
- 请求侧 Turnstile 中间件，已支持 Cloudflare Turnstile 服务端校验。
- 请求侧统一认证与 API Key 认证，可区分匿名流量与认证流量。
- 请求侧 Redis 限流中间件，可按 IP、按用户限流，并支持按服务维度覆盖配置。
- OPA + Casbin 混合授权能力，可承载风险策略、灰度放行和审计逻辑。

现阶段的主要缺口有两个：

- Bot 管理、代理识别、机房识别并不是网关内建能力，需要依赖 CDN/WAF 或外部情报源提供风险标签。
- 目前缺少专门的风险处置中间件，尚未把 challenge、observe、block、allow 这类动作抽象成统一策略层。

## 模块映射

| 安全需求 | 主要承载模块 | 说明 |
|------|------|------|
| 收紧 CORS 白名单 | `CorsMiddleware`、`MiddlewareConfigurator` | 当前已经支持白名单模式，但默认仍挂载 permissive 配置 |
| Bot 管理与基础风控 | `ClientContextMiddleware`、`RateLimitMiddleware`、`AuthorizationService` | 接收 CDN/WAF 风险头，结合限流与策略引擎执行二次处置 |
| 匿名高价值入口人机验证 | `TurnstileMiddleware`、`TurnstileConfig` | 当前已有完整实现，确定统一使用 Cloudflare Turnstile |
| 异常国家、机房、代理流量灰度拦截 | `ClientContextMiddleware`、`AuthorizationService`、`RateLimitMiddleware` | 识别结果来自上游，网关负责消费标签并执行灰度动作 |

## 总体方案

### 1. 收紧 CORS 白名单

建议将 CORS 从当前的全开放模式调整为按环境、按域名白名单控制。

推荐做法：

- 生产环境只允许受信任的前端域名，例如主站、管理后台、合作方回调域名。
- 测试环境与开发环境使用独立白名单，避免把测试域名带入生产。
- 对需要携带 Cookie 或认证头的前端域名启用 credentials，并禁止 `*`。
- 对外开放 API 和内部管理 API 使用不同的 CORS 配置，避免后台接口被前端跨域访问。

落地方式：

- 将 `MiddlewareConfigurator` 中默认挂载的 permissive 模式改为按服务读取 ETCD 中的 CORS 白名单。
- 在 `ServiceInstance.middleware_config` 中为对应服务增加 `cors_enabled` 和 `cors` 配置。
- 对 `/_admin` 等管理接口进一步收紧，建议仅内网访问，并配合反向代理 ACL。

预期收益：

- 降低浏览器侧跨域误暴露风险。
- 减少被第三方站点直接调用受保护接口的机会。

### 2. 接入 CDN/WAF Bot 管理与基础风控

建议将 Bot 管理分成两层：

- 第一层由 CDN/WAF 负责识别和打标。
- 第二层由网关负责接收标签并进行二次决策。

推荐的上游输入包括：

- 客户端真实 IP。
- 国家、地区、ASN、数据中心、代理/VPN、Tor、Bot 分数等风险字段。
- 已通过或已触发的 WAF/Bot 规则标识。

网关侧建议：

- 在 `ClientContextMiddleware` 中统一提取并标准化这些头，写入 `x-client-context`。
- 保留原始风险头透传能力，便于后端审计与问题排查。
- 在 OPA/Casbin 中定义策略规则，例如高风险 Bot 直接拒绝、中风险 Bot 要求人机验证、低风险 Bot 仅加强限流。
- 在 `RateLimitMiddleware` 上叠加更严格的匿名限流与热点路径限流。
- 风险中间件开关与规则按注册服务写入 ETCD，而不是全局统一启停。

边界说明：

- Bot 管理引擎不建议在网关内重复实现。
- 网关更适合做策略编排、风险汇总、二次挑战和审计。

### 3. 匿名高价值入口统一启用 Cloudflare Turnstile

本方案明确将人机验证统一收敛为 Cloudflare Turnstile。

适用入口：

- 登录。
- 注册。
- 找回密码。
- 验证码发送。
- 邀请码校验。
- 匿名表单提交。
- 其他高价值且容易被脚本滥用的匿名入口。

接入原则：

- Turnstile 位于认证前，验证失败直接拒绝，不进入后续认证和业务处理。
- Secret Key 仅保存在网关服务端。
- 前端负责渲染 Widget 并把 token 放入请求头。
- 业务服务默认无需感知 Turnstile，仅由网关统一验证。
- 是否对某服务启用 Turnstile 以及哪些路径强制 challenge，按服务写入 ETCD。

推荐策略：

- 默认仅对匿名高价值入口启用，而不是全站强制开启。
- 对同一入口可结合路径白名单和 action 校验，避免 token 被跨页面复用。
- 对反复验证失败的来源叠加限流和短时封禁。

与现有能力的关系：

- 当前网关已经具备 Turnstile 中间件与配置结构。
- 现阶段重点应放在按路径精细启用、与风控策略联动、补齐监控与审计。

### 4. 异常国家、机房、代理流量灰度拦截

建议把这类控制做成“灰度处置”而不是单纯二元封禁。

推荐动作分级：

- `allow`：正常放行。
- `observe`：只记录日志和指标，不影响请求。
- `throttle`：更严格限流。
- `challenge`：要求通过 Cloudflare Turnstile。
- `block`：直接拒绝。

推荐判定条件：

- 非业务覆盖国家或异常国家来源。
- 已知云厂商机房、托管机房、匿名代理、VPN、Tor 节点。
- 短时间内跨账号撞库、批量注册、验证码轰炸等行为模式。

推荐落地方式：

- 风险识别结果由 CDN/WAF 或外部情报源提供。
- `ClientContextMiddleware` 负责提取国家、地区和风险头。
- 新增风险处置中间件，或者在授权策略层中实现灰度决策。
- 高风险流量先 challenge，再根据结果决定放行或封禁。

不建议直接做法：

- 不建议仅凭国家字段直接永久封禁。
- 不建议把所有机房流量一刀切拒绝，优先采用观察、加强限流、人机验证三阶段策略。

## 推荐的 CDN/WAF 与情报源选型

### 推荐选型原则

建议优先选择能够同时满足以下条件的服务：

- 能在边缘侧直接提供 WAF、Bot 管理、限流和规则引擎能力。
- 能向源站稳定透传国家、地区、ASN、数据中心、代理、Bot 分数等标签。
- 能与 Cloudflare Turnstile 协同工作，避免重复接入多套人机验证体系。
- 能被网关中间件和策略层稳定消费，而不是只能在控制台中孤立使用。

### 第一优先级：CDN/WAF 一体化平台

#### Cloudflare

最推荐作为当前方案的主选型。

适用原因：

- 已确定人机验证统一使用 Cloudflare Turnstile，生态一致性最好。
- 同一体系内可提供 CDN、WAF、Bot Management、Rate Limiting、Rules、地理与 ASN 识别。
- 边缘规则和源站头透传都比较成熟，适合当前网关消费风险标签。

适合作为：

- 主 CDN/WAF。
- 主 Bot 管理平台。
- 主人机挑战平台。

#### 阿里云 WAF / 腾讯云 WAF

适合以中国大陆网络环境为主的部署场景。

适用原因：

- 国内接入、线路和备案配套更现实。
- 能提供基础 WAF、地域和来源控制、部分 Bot 风控能力。
- 更适合作为国内业务的边界防护层。

适合作为：

- 国内业务主 WAF。
- Cloudflare 不作为主入口时的替代方案。

#### Akamai / Imperva / F5 Distributed Cloud WAF

适合已有企业安全体系或高预算场景。

适用原因：

- Bot 对抗和企业级安全治理能力较强。
- 更适合大型企业、强合规或已有既有平台投入的环境。

适合作为：

- 企业级替代方案。
- 已有成熟安全团队运营的场景。

### 第二优先级：外部情报源

#### MaxMind GeoIP2

适合补充国家、地区、ASN、机房和匿名代理基础识别。

适合作为：

- GeoIP 基础源。
- 国家、地域、ASN、托管机房判断的补充情报源。

#### IPinfo

适合快速补充 ASN、Company、Hosted、Privacy 等字段。

适合作为：

- 接口式风险补充源。
- 对 ASN、托管、运营商、隐私属性有较强需求的场景。

#### IP2Location

适合做 GeoIP 和代理基础识别的替代方案。

适合作为：

- MaxMind 的替代或补充源。

#### Spur / IPQualityScore

适合识别代理、VPN、住宅代理、Tor、自动化流量风险。

适合作为：

- 代理/VPN/Tor 风险识别源。
- 机房和匿名网络灰度拦截的补充判断源。

### 第三优先级：专业 Bot/反自动化平台

#### DataDome / HUMAN / Kasada / Arkose Labs

适合在撞库、批量注册、爬虫、工作室脚本对抗压力很高时使用。

适用原因：

- 专业 Bot 和自动化流量识别能力通常强于基础 CDN/WAF。
- 更适合高对抗、高滥用风险的匿名入口和营销场景。

注意事项：

- 这类平台通常价格和接入复杂度更高。
- 如果当前已经使用 Cloudflare Turnstile，应避免同时引入多套前端挑战体系，优先让专业平台输出风险标签，由网关决定是否继续使用 Turnstile challenge。

### 推荐组合

根据当前方案，推荐按优先级考虑以下组合：

1. Cloudflare + Turnstile。
适合希望以最少接入成本完成 CDN、WAF、Bot 和人机验证闭环的场景。

2. Cloudflare + MaxMind。
适合在 Cloudflare 基础上补强国家、ASN、数据中心、GeoIP 一致性的场景。

3. Cloudflare + Spur 或 IPQualityScore。
适合重点识别代理、VPN、Tor、住宅代理风险的场景。

4. 阿里云 WAF 或 腾讯云 WAF + MaxMind/IP2Location。
适合国内业务为主、但仍需要外部 GeoIP/代理识别补充的场景。

## 推荐的中间件顺序

建议将安全相关处理顺序明确为：

```text
RequestId
-> RequestIdSpan
-> ClientContext
-> RiskAssessment（新增，消费 CDN/WAF 风险标签）
-> Turnstile（仅命中 challenge 路径或策略时启用）
-> UnifiedAuth
-> RateLimit
-> HttpToGrpcMetadata
-> Logging
-> Proxy / Local Service
```

顺序理由：

- 先提取客户端上下文，后续策略才能使用真实来源信息。
- 风险判断应早于认证和业务逻辑，以便匿名流量先被 challenge 或拒绝。
- 人机验证要早于认证，避免机器人消耗认证服务资源。
- 限流位于认证之后，便于同时对匿名 IP 和登录用户实施差异化控制。

## 实施清单

以下实施清单按三个阶段展开，分别对应配置改造、中间件改造、外部 CDN/WAF 对接。建议按顺序推进，先完成低风险收敛，再做策略化增强。

### 第一阶段：配置改造

目标：先用最小代码改动完成白名单收敛、入口保护和基础风控开关准备。

#### 1. CORS 配置收敛

- 增加 CORS 白名单配置项，覆盖 `allowed_origins`、`allowed_methods`、`allowed_headers`、`allow_credentials`。
- 区分生产、测试、开发三套域名白名单，避免测试域名误入生产。
- 管理接口和公开 API 使用不同白名单，`/_admin` 默认仅允许内网或堡垒机来源。
- 明确是否允许携带 Cookie；若允许，则禁止使用 `*`。

#### 2. Turnstile 配置收敛

- 在所有匿名高价值入口清点路径清单。
- 为登录、注册、验证码发送、找回密码等入口定义统一 challenge 路径配置。
- 配置 `turnstile.enabled`、`token_header`、`expected_action`、`expected_hostname`。
- 区分开发、测试、生产环境的 Site Key 和 Secret Key 管理方式。

#### 3. 风控配置预埋

- 在 `ServiceInstance.middleware_config` 中扩展服务级风险配置，至少覆盖 `risk_enabled`、`challenge_paths`、`high_risk_countries`、`challenge_proxy_traffic`、`block_datacenter_traffic`、`bot_score_threshold`。
- 明确是否信任 `cf-connecting-ip`、`x-forwarded-for`、`cf-ipcountry` 等代理头。
- 定义各环境风险阈值，避免测试环境沿用生产拦截阈值。

#### 4. 审计与监控准备

- 为 challenge、block、observe、throttle 等动作定义日志字段。
- 预留指标名称和标签，例如国家、ASN、代理类型、Bot 分段、命中规则。
- 确认日志和监控平台能接收网关输出的安全事件。

阶段验收标准：

- CORS 不再使用默认全开放模式。
- Turnstile 目标路径清单和配置项明确。
- 风控配置结构落地，可支持后续中间件实现。

### 第二阶段：中间件改造

目标：让网关具备消费风险标签、执行灰度动作和联动 Turnstile 的能力。

#### 1. ClientContext 扩展

- 扩展 `ClientContextMiddleware`，统一提取国家、地区、ASN、数据中心、代理、VPN、Tor、Bot 分数、WAF 命中标记等字段。
- 为不同厂商的头建立统一字段映射，避免策略层直接依赖私有头名。
- 保留原始头信息或原始标签，便于问题排查和审计。

#### 2. 风险处置中间件

- 新增 `RiskAssessmentMiddleware`，位置放在 `ClientContext` 之后、`Turnstile` 之前。
- 根据风险标签输出统一动作：`allow`、`observe`、`throttle`、`challenge`、`block`。
- 对匿名入口优先执行 challenge，再决定是否进入认证。
- 对已认证请求保留更细粒度处置能力，例如加强限流而非直接阻断。

#### 3. Turnstile 联动

- 让风险处置中间件可以决定是否强制要求 Turnstile。
- 支持“路径命中要求 challenge”和“风险命中要求 challenge”两种模式。
- 为 challenge 失败、重复失败、token 缺失设计统一错误和日志格式。

#### 4. RateLimit 联动

- 为高风险匿名流量应用更严格 IP 限流。
- 对 challenge 失败来源设置更严格的临时阈值。
- 对高价值入口单独设置热点路径阈值，而不是只靠全局限流。

#### 5. OPA/Casbin 策略联动

- 将风险标签、用户状态、路径类型、来源信息传入策略层。
- 由 OPA/Casbin 管理 `allow`、`observe`、`challenge`、`block` 的决策条件。
- 为后台入口、匿名入口、普通用户入口设置不同阈值和策略。

#### 6. 审计与指标输出

- 记录每次风险动作的命中原因、输入标签、最终动作。
- 输出 challenge 命中率、challenge 通过率、block 命中率、代理流量命中量、国家拦截量等指标。
- 保证审计日志可关联 request_id、session_id、user_id、IP、国家、Bot 分数。

阶段验收标准：

- 网关可以消费上游风险标签并输出统一动作。
- 高风险流量能够被 challenge、throttle 或 block，而不是只能记录日志。
- Turnstile、限流、策略层三者联动可观测。

### 第三阶段：外部 CDN/WAF 对接

目标：让边缘识别能力和网关策略执行形成闭环。

#### 1. 选型确认

- 选择主 CDN/WAF 平台，优先推荐 Cloudflare。
- 如果业务主要在中国大陆，评估阿里云 WAF 或腾讯云 WAF 作为边界入口。
- 确认是否需要额外接入 GeoIP、代理识别或专业 Bot 情报源。

#### 2. 头部与标签对接

- 梳理 CDN/WAF 能透传的国家、地区、ASN、数据中心、代理、Bot 分数、规则命中头。
- 确认源站与网关之间只信任来自 CDN/WAF 的风险头，避免客户端伪造。
- 在反向代理或网关入口处做头部清洗，覆盖同名客户端头。

#### 3. 边缘规则配置

- 在 CDN/WAF 配置基础拦截规则，例如已知恶意国家、明显恶意 UA、已知攻击特征。
- 将中风险流量透传给网关处理，不在边缘全部封死。
- 对高风险匿名入口优先使用边缘识别加网关联动 challenge，而不是只靠单一侧决策。

#### 4. 情报源补充

- 若主 CDN/WAF 不能稳定提供 ASN、数据中心、代理/VPN/Tor 标签，补接 MaxMind、IPinfo、IP2Location、Spur、IPQualityScore 等服务。
- 为外部情报源设计缓存策略和降级策略，避免实时查询拖慢请求链路。
- 明确情报源失效时的默认行为，建议优先降级到 `observe` 或基础限流，而不是全部放行或全部阻断。

#### 5. 闭环验证

- 用真实测试样本验证国家、机房、代理、Bot 分数标签是否能到达网关。
- 验证边缘规则命中后，网关是否还能正确识别 challenge 或 block 结果。
- 验证日志、指标、审计链路是否能串起边缘和网关两侧事件。

阶段验收标准：

- CDN/WAF 输出的风险头能被网关稳定消费。
- 高风险流量在边缘和网关之间有明确分工，不出现重复挑战或策略冲突。
- 外部情报源失效时，网关具备明确降级路径。

## 配置与实现建议

### 建议新增或补齐的配置项

```yaml
security:
  cors:
    allowed_origins:
      - "https://app.example.com"
      - "https://admin.example.com"
    allowed_methods: ["GET", "POST", "PUT", "DELETE", "OPTIONS"]
    allowed_headers: ["authorization", "content-type", "x-request-id"]
    allow_credentials: true

  risk:
    trust_proxy_headers: true
    challenge_paths:
      - "/auth/login"
      - "/auth/register"
      - "/auth/send-code"
    high_risk_countries: ["XX"]
    block_datacenter_traffic: false
    challenge_proxy_traffic: true
    bot_score_threshold: 30

turnstile:
  enabled: true
  token_header: "cf-turnstile-response"
  expected_action: "login"
```

说明：

- `turnstile` 继续沿用现有配置结构。
- CORS、风险和 Turnstile 路径规则建议挂到 `ServiceInstance.middleware_config`，并随服务注册写入 ETCD。
- 平台级基础配置只保留可信代理头、Turnstile 密钥和底层客户端参数。

### 建议新增的网关能力

- 风险处置中间件：统一根据来源国家、机房、代理、Bot 分数决定放行、挑战、封禁。
- 风险标签标准化：把不同 CDN/WAF 的头映射为统一字段，避免策略层直接依赖厂商私有头名。
- 审计与指标：记录 challenge 命中率、challenge 通过率、国家拦截量、代理流量命中量、Bot 分数分布。

## 推荐的最终落地形态

生产环境建议形成如下分层：

1. CDN/WAF：负责 IP 信誉、Bot 识别、国家与机房识别、基础拦截。
2. Stew 网关：负责 ClientContext 汇总、Cloudflare Turnstile 验证、统一认证、限流、风险二次决策、审计。
3. 业务服务：只处理业务权限与业务风控，不重复实现边界安全逻辑。

这样可以把识别、挑战、鉴权、审计分层清楚，降低业务服务重复接入成本。

---

## 结论

对于当前 Stew 网关，推荐的改进方向不是新增一套独立安全网关，而是在现有中间件链上补齐三件事：

- 把 CORS 从默认放开改为白名单配置化。
- 把匿名高价值入口统一接入 Cloudflare Turnstile。
- 把 CDN/WAF 输出的风险标签接入 ClientContext，并通过授权与限流能力完成灰度处置。

其中，人机验证方案确定统一使用 Cloudflare Turnstile；国家、机房、代理、Bot 风险识别则建议交给上游 CDN/WAF，Stew 网关负责策略执行、挑战联动和审计闭环。
