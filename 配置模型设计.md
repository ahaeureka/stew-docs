# 配置模型设计

> 返回 [README](README.md) | 参阅 [配置手册](配置手册.md) | [安全增强方案](安全增强方案.md)

---

## 目标

为 Stew 网关新增一组可落地的安全配置模型，用于承载以下能力：

1. CORS 白名单配置化。
2. 风险识别与风险处置配置化。
3. Cloudflare Turnstile 与风险挑战联动。
4. 外部 CDN/WAF 与情报源风险标签消费。

本设计文档关注配置结构本身，不直接实现业务逻辑。

本版设计以按注册服务绑定配置为前提：

- 配置不再以网关全局 `AppConfig.security` 为主。
- 配置逻辑上绑定到服务实例本身。
- 配置随服务注册或编辑一起写入 ETCD。
- 网关通过 ETCD watch 热更新对应服务的安全中间件开关和参数。

当前实现决策：

- 已在 `ServiceMiddlewareConfig` 中补充正式的 `cors`、`risk`、`turnstile` 字段及开关。
- 运行时优先读取 `middleware_config` 中的正式字段。
- `ServiceInstance.metadata.stew_security` 继续保留为兼容镜像，用于平滑迁移历史数据和旧页面调用。

---

## 设计原则

- 优先复用现有 `ServiceInstance.middleware_config` 扩展点，避免再引入新的全局运行时配置源。
- 新增配置应支持 YAML 和环境变量两种来源。
- 对现有 `turnstile`、`rate_limit` 配置尽量复用，不重复定义。
- 配置模型要优先支持服务级开关、路径级覆盖和规则级覆盖。
- 未配置时应有安全可控的默认行为，不影响现有系统启动。

补充说明：

- `AppConfig` 中可以保留默认值或回退值。
- 真正生效的安全策略以 ETCD 中服务实例上的配置为准。

---

## 当前配置现状

当前 `AppConfig` 已包含以下安全相关配置：

- `server.skip_auth_paths`
- `rate_limit`
- `turnstile`
- `authorization`

当前缺少的配置能力主要有：

- CORS 白名单的结构化配置。
- 风险标签信任来源配置。
- 风险动作配置，如 challenge、block、observe、throttle。
- 不同路径或服务对风险控制的覆盖规则。

当前仓库里已经存在一个关键能力：

- `ServiceInstance.middleware_config` 已经会随服务注册写入 ETCD。
- `RateLimitMiddleware` 已经支持从 ETCD 中读取按服务配置覆盖全局值。

因此安全增强应沿用同一模式，而不是再引入一套新的全局配置主入口。

---

## 推荐的配置承载位置

从目标模型上，新增安全配置应承载在服务实例上，而不是放在 `AppConfig` 顶层。

结合当前仓库的生成代码成本，现阶段采用以下分层：

- 逻辑归属：服务实例级配置。
- 正式存储位置：`ServiceInstance.middleware_config`。
- 兼容镜像位置：`ServiceInstance.metadata.stew_security`。

推荐关系如下：

- `AppConfig.rate_limit`：全局默认值和回退值。
- `AppConfig.turnstile`：Turnstile 客户端能力和站点级默认值。
- `ServiceInstance.middleware_config`：某个注册服务的安全开关和规则。

也就是说：

- 是否对某服务启用风险中间件，是服务级配置。
- 是否对某服务启用 CORS 白名单收敛，是服务级配置。
- 某服务哪些路径需要 challenge，是服务级配置。
- 配置写入 ETCD 后，由网关 watch 并热生效。

---

## Proto 演进决策

本轮已经完成 proto 正式化，但采用双写兼容策略，而不是一次性切换：

- 新写入请求同时更新 `middleware_config` 和 `metadata.stew_security`。
- 网关运行时优先读 `middleware_config`，若未配置再回退到 metadata。
- 历史 ETCD 数据无需一次性回填，也不会阻断当前热更新链路。

这样做的目的：

- 对外提供强类型的服务注册/编辑契约。
- 避免旧实例、旧页面或历史数据在迁移窗口内失效。
- 为后续完全移除 metadata 载体预留灰度周期。

## 推荐的 proto 扩展方向

当前 `ServiceMiddlewareConfig` 已包含：

- `rate_limit_enabled`
- `rate_limit_rpm`
- `rate_limit_user_rpm`

当进入正式 schema 迁移阶段时，建议在同一个消息体上继续扩展安全相关字段：

```protobuf
message ServiceMiddlewareConfig {
  bool rate_limit_enabled = 1;
  uint32 rate_limit_rpm = 2;
  uint32 rate_limit_user_rpm = 3;

  bool cors_enabled = 10;
  ServiceCorsConfig cors = 11;

  bool risk_enabled = 20;
  ServiceRiskConfig risk = 21;

  bool turnstile_enabled = 30;
  ServiceTurnstileConfig turnstile = 31;
}
```

这样做的好处：

- 服务注册接口无需新增新的存储路径。
- 服务编辑页面可以直接复用已有 `middleware_config` 编辑入口。
- ETCD 中服务实例和安全策略天然保持一致。
- 网关热更新链路与现有限流实现一致。

## 推荐的配置模型

### 1. ServiceCorsConfig

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Default)]
#[serde(default)]
pub struct ServiceCorsConfig {
  pub allow_origins: Vec<String>,
  pub allow_methods: Vec<String>,
  pub allow_headers: Vec<String>,
  pub expose_headers: Vec<String>,
  pub allow_credentials: bool,
  pub max_age_secs: u32,
  pub path_overrides: Vec<ServiceCorsRuleConfig>,
}
```

职责：

- 定义某个注册服务自己的 CORS 策略。
- 不同服务可以拥有不同来源白名单。

### 2. ServiceCorsRuleConfig

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(default)]
pub struct ServiceCorsRuleConfig {
  pub name: String,
  pub path_prefixes: Vec<String>,
    pub allow_origins: Vec<String>,
  pub allow_credentials: Option<bool>,
}
```

建议默认值：

- `allow_origins = []`
- `allow_methods = ["GET", "POST", "PUT", "DELETE", "PATCH", "OPTIONS"]`
- `allow_headers = ["authorization", "content-type", "x-request-id"]`
- `expose_headers = ["content-type", "content-length", "date", "x-request-id"]`
- `allow_credentials = false`
- `max_age_secs = 3600`

设计说明：

- 空 `allow_origins` 表示该服务默认不允许任意跨域，必须显式配置。
- `path_overrides` 用于同一服务下的管理路径或高敏路径单独收敛。

### 3. TrustedProxyConfig

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(default)]
pub struct TrustedProxyConfig {
    pub trust_proxy_headers: bool,
    pub trusted_cdn_ips: Vec<String>,
    pub trusted_proxy_ips: Vec<String>,
    pub client_ip_headers: Vec<String>,
    pub country_headers: Vec<String>,
    pub region_headers: Vec<String>,
    pub asn_headers: Vec<String>,
    pub datacenter_headers: Vec<String>,
    pub proxy_headers: Vec<String>,
    pub bot_score_headers: Vec<String>,
    pub waf_rule_headers: Vec<String>,
}
```

设计说明：

- 控制网关信任哪些上游头，而不是无条件接受客户端传来的同名头。
- 为多 CDN/WAF 场景预留头部优先级顺序。
- 这是少数仍然建议保留在网关侧的基础配置。
- 但具体是否消费这些风险头，仍由服务级 `risk_enabled` 决定。

### 4. ServiceRiskConfig

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(default)]
pub struct ServiceRiskConfig {
    pub mode: RiskMode,
    pub default_action: RiskAction,
    pub challenge_paths: Vec<String>,
    pub block_paths: Vec<String>,
    pub observe_only_paths: Vec<String>,
    pub high_risk_countries: Vec<String>,
    pub challenge_proxy_traffic: bool,
    pub block_datacenter_traffic: bool,
    pub allow_tor_exit_nodes: bool,
    pub bot_score_threshold: u32,
    pub proxy_score_threshold: u32,
    pub action_overrides: Vec<ServiceRiskRuleConfig>,
}
```

其中：

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Default)]
pub enum RiskMode {
    #[default]
    Disabled,
    Observe,
    Enforce,
}

#[derive(Debug, Clone, Serialize, Deserialize, Default)]
pub enum RiskAction {
    #[default]
    Allow,
    Observe,
    Throttle,
    Challenge,
    Block,
}
```

设计说明：

- `mode` 控制该服务的运行模式，方便按服务灰度上线。
- `default_action` 是无匹配规则时的回退动作。
- `action_overrides` 支持更细粒度规则覆盖。

### 5. ServiceRiskRuleConfig

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Default)]
#[serde(default)]
pub struct ServiceRiskRuleConfig {
    pub name: String,
    pub enabled: bool,
    pub path_prefixes: Vec<String>,
    pub service_names: Vec<String>,
    pub countries: Vec<String>,
    pub asns: Vec<String>,
    pub datacenter: Option<bool>,
    pub proxy: Option<bool>,
    pub vpn: Option<bool>,
    pub tor: Option<bool>,
    pub min_bot_score: Option<u32>,
    pub action: RiskAction,
}
```

设计说明：

- 用于路径、服务、来源属性组合判断。
- 适合作为 `RiskAssessmentMiddleware` 的静态规则输入。

### 6. ServiceTurnstileConfig

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Default)]
#[serde(default)]
pub struct ServiceTurnstileConfig {
  pub required_paths: Vec<String>,
  pub skip_paths: Vec<String>,
  pub expected_action: Option<String>,
  pub expected_hostname: Option<String>,
  pub enforce_on_risk_challenge: bool,
}
```

设计说明：

- Turnstile 密钥、HTTP 客户端超时仍可保留在全局。
- 但某个服务是否启用 challenge、哪些路径要 challenge，应跟着服务实例走。

## ETCD 中的配置形态示例

推荐不是把这些策略单独写入 YAML 文件长期维护，而是体现在注册到 ETCD 的 `ServiceInstance.middleware_config` 中。

可以对应为注册请求中的结构：

```json
{
  "service_name": "stew.api.v1.AuthService",
  "instance_id": "auth-1",
  "middleware_config": {
    "rate_limit_enabled": true,
    "rate_limit_rpm": 120,
    "rate_limit_user_rpm": 30,
    "cors_enabled": true,
    "cors": {
      "allow_origins": ["https://app.example.com"],
      "allow_methods": ["GET", "POST", "OPTIONS"],
      "allow_headers": ["authorization", "content-type", "x-request-id"],
      "expose_headers": ["content-type", "x-request-id"],
      "allow_credentials": true,
      "max_age_secs": 3600,
      "path_overrides": [
        {
          "name": "admin-login",
          "path_prefixes": ["/auth/admin"],
          "allow_origins": ["https://admin.example.com"],
          "allow_credentials": true
        }
      ]
    },
    "risk_enabled": true,
    "risk": {
      "mode": "enforce",
      "default_action": "allow",
      "challenge_paths": ["/auth/login", "/auth/register"],
      "high_risk_countries": ["XX"],
      "challenge_proxy_traffic": true,
      "block_datacenter_traffic": false,
      "allow_tor_exit_nodes": false,
      "bot_score_threshold": 30,
      "proxy_score_threshold": 70,
      "action_overrides": [
        {
          "name": "proxy-login",
          "enabled": true,
          "path_prefixes": ["/auth/login"],
          "proxy": true,
          "action": "challenge"
        }
      ]
    },
    "turnstile_enabled": true,
    "turnstile": {
      "required_paths": ["/auth/login", "/auth/register"],
      "skip_paths": ["/health"],
      "expected_action": "login",
      "enforce_on_risk_challenge": true
    }
  }
}
```

## 配置写入与分发方式

推荐写入路径：

- 服务注册接口 `RegisterService`
- 服务编辑接口 `UpdateServiceInstance`
- 管理后台服务编辑页中的 `middleware_config`

推荐分发链路：

1. 控制面创建或编辑 `ServiceInstance.middleware_config`。
2. 服务实例整体写入 ETCD 的 `/services/{service_name}/{instance_id}`。
3. `EtcdServiceDiscovery` watch 到变更并更新本地缓存。
4. 对应中间件按 `service_name` 读取最新配置并热生效。

这个模式与当前 `RateLimitMiddleware` 已有实现一致。

## 与现有配置的关系

### 兼容保留

- `turnstile` 顶层结构保留。
- `rate_limit` 顶层结构保留。
- `server.skip_auth_paths` 保留，不与服务级 `risk.challenge_paths` 冲突。

### 迁移建议

- 现有 `CorsMiddleware::permissive()` 改为优先读取服务实例上的 `middleware_config.cors`。
- `ClientContextMiddleware` 仍读取平台级 `trusted_proxy`，决定信任哪些头。
- 新增 `RiskAssessmentMiddleware` 读取服务实例上的 `middleware_config.risk`。
- `TurnstileMiddleware` 继续读取全局密钥配置，但是否执行由服务实例上的 `middleware_config.turnstile` 决定。

---

## 实现顺序建议

1. 在 `service_discovery.proto` 中扩展 `ServiceMiddlewareConfig`。
2. 更新服务注册和服务编辑接口的请求结构与前端表单。
3. 让 `CorsMiddleware` 按 `service_name` 从 ETCD 缓存读取 `middleware_config.cors`。
4. 让 `RiskAssessmentMiddleware` 按 `service_name` 从 ETCD 缓存读取 `middleware_config.risk`。
5. 让 `TurnstileMiddleware` 按 `service_name` 读取 `middleware_config.turnstile` 决定是否执行 challenge。
6. 仅把 `trusted_proxy` 这类平台级基础能力保留在全局配置中。

---

## 验收标准

- 新配置模型以服务实例为粒度存储在 ETCD 中。
- 不同注册服务可拥有不同的 CORS、风险、Turnstile 开关和参数。
- ETCD 中服务实例配置变更后无需重启网关即可生效。
- 未配置扩展安全字段的服务保持与当前版本兼容。
